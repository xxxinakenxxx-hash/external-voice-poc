# AGENTS.md

このファイルは、Codex がこのプロジェクト（営業AI音声メモ）で作業する際に常に従う「変わらないルール」を定めます。

最新の業務方針・確定事項・文書の版数は、このファイルではなく、次を参照してください。

- 共通運用規約：`docs/ルールブック.md`（三者共通の運用規約。本ファイルより上位）
- 現在地・直近の確定事項・次の作業：`tasks/TASK_CURRENT.md`
- 仕様・設計の正：`docs` 内の正式文書（Word/Excel）

音声入力・GPS・店舗マスタ照合・OpenAI・Googleログイン等の「変わりうる方針」は、このファイルに書きません（古くなって事故の原因になるため）。これらは正式文書と `tasks/TASK_CURRENT.md` を正とします。

---

## 全工程共通で必ず読むもの

どの工程の作業でも、着手前に必ず次の3つを読みます。

1. `tasks/TASK_CURRENT.md` … 現在地・確定事項・次の作業
2. `docs/ルールブック.md` … 三者共通の運用規約（最上位）
3. `AGENTS.md`（本ファイル） … Codex の不変ルール

そのうえで、**その工程で読むファイル（DB_SCHEMA.md / SCREEN_FLOW.md / API_INTERNAL.md や関係する正式文書など）は、対象工程の実装指示書が指定します。** 実装指示書が「読むもの」と「変更してよい対象」を指定するので、その指定に従ってください。実装指示書が存在しない場合は、実装を始めず「実装指示書未作成」と報告して停止します。

---

## 0. 体制と立ち位置

このプロジェクトは、Google Apps Script（GAS）による「営業AI音声メモ（営業メモ音声入力システム＋メモ活用）」です。作業は次の4者で分担します。Codex は実装担当です。

- 監督（けんたろう）：目的・仕様判断・承認・差戻し・画面目視確認・デプロイ判断。
- 設計主幹（Claude）：正本文書・基本設計・開発工程表・実装指示書を作る。
- 工程管理（ChatGPT）：工程順を管理し、Codex への作業指示（プロンプト）を作り、報告を読み解く。`tasks/TASK_CURRENT.md` を管理する。
- 実装（Codex＝あなた）：承認された設計・指示だけをコードと実ファイルへ反映する。仕様も設計も考えない。

Codex は、業務仕様や設計を自分で決めてはなりません。決まっていないことに出会ったら、止めて報告します。

---

## 1. 実装前の確認

作業前に、次の順で確認してください。

1. 「全工程共通で必ず読むもの」の3つ（`tasks/TASK_CURRENT.md` / `docs/ルールブック.md` / 本ファイル）
2. 対象工程の実装指示書を開く
3. 実装指示書が「読むもの」として指定した文書・ファイル（DB_SCHEMA.md / SCREEN_FLOW.md / API_INTERNAL.md や関係する正式文書など）を、指定された範囲で読む
4. 実装指示書が指定した「変更してよい対象」（既存の `src` 構成・シート名・列名・関数名・画面名など）を確認する

対象工程の実装指示書が存在しない場合は、開発工程表や仕様書だけを根拠に実装を始めず、「実装指示書未作成」と報告して停止してください。工程ごとに読むファイルは、このAGENTSではなく実装指示書が指定します。

---

## 2. 事前報告と承認

ファイルを変更する前に、必ず次をユーザーへ報告してください。

- 変更予定ファイル
- 修正方針
- 影響範囲
- 確認した関連文書
- 不明点・未確定事項・要確認事項
- 実装しない範囲

ユーザーから明示的な承認を得るまで、ファイルの作成・編集・削除・移動・名前変更を行わないでください。

---

## 3. 変更時の原則

変更は最小差分で行ってください。

- 依頼範囲外のリファクタリングを行わない
- 仕様外の画面・機能を追加しない
- 既存のシート名・列名・関数名・画面名を勝手に変更しない
- 既存データを破壊する処理を追加しない
- 既存の入力形式・出力形式・保存形式を無断で変更しない
- 不明な仕様は推測で実装せず、要確認として報告する
- 「良かれと思って」の追加作業をしない

---

## 4. GAS構成・appsscript.json の扱い

このアプリは、`doGet(e)` をエントリとする GAS Webアプリです。画面はSPA方式（`doGet` は初回アクセス時のみ呼ばれ、画面切替はクライアント側のDOM操作で行う）で構成します。URLパラメータは使用しません。

`appsscript.json` は、スコープ・タイムゾーン・実行設定・Webアプリ設定などに影響します。特に本アプリは「オーナーとして実行」を前提とします。ユーザーの明示的な承認がない限り、`appsscript.json` を変更しないでください。

---

## 5. データ保護

既存データを破壊する可能性がある処理は禁止します。特に次は、事前に個別承認を得てください。

- シートの削除
- 行や列の一括削除
- セル範囲の一括クリア
- 既存データの上書き
- データ構造の変更
- 初期化処理・マイグレーション処理
- テストデータ投入による既存データへの影響

商談記録シートは追記（新規行追加）専用です。過去レコードを更新・上書きしないでください。

---

## 6. clasp・デプロイ・外部通信

次の操作は、通常の実装承認とは別に、個別の明示的承認を得てから行ってください。

- clasp 操作、GAS への push
- デプロイ、Webアプリ公開設定の変更
- 外部サービス連携設定の変更
- パッケージのインストール
- 設定ファイルの変更

GAS への反映は、必ず `clasp push --force` と表記します。`clasp push` / `push` / `claspで反映` 等の省略表記は用いません（`docs/ルールブック.md` §8）。

このプロジェクトのコード反映経路は「デスクトップ版Codexで編集 → ローカルで編集を確定 → `clasp push --force` でGASへ反映」です。GAS スクリプトエディタでの直接編集、および GitHub 経由の中継は行いません。

Codex が実施してよいのは、静的チェックOK後の `clasp push --force` までです。デプロイ更新と実画面確認（`/exec`）は監督（けんたろう）の作業であり、Codex は完了扱いにしないでください。

---

## 7. docs フォルダの扱い

`docs` 内の文書群は、このプロジェクトの正式な仕様・設計・工程管理資料です。実装前に、対象工程に関係する開発工程表・仕様書・設計書・未確定事項を確認してください。

`docs` 内の Markdown（PROJECT_OVERVIEW.md / SPEC_WEBAPP.md / DB_SCHEMA.md / SCREEN_FLOW.md / API_INTERNAL.md 等）は、正式文書（Word/Excel）を Codex 実装用に抜粋したものです。Markdown だけを根拠に実装せず、列定義・関数・画面等の最終的な正は正式文書側を確認してください。

文書間に矛盾がある場合は、独断で判断せず、矛盾点を報告して停止してください。どちらの文書を採用するかを Codex が決めてはなりません。

---

## 8. 仕様判断の優先順位

仕様判断は、次の優先順位で行ってください。

1. ユーザーからの明示的な最新指示
2. `tasks/TASK_CURRENT.md` の現在地・確定事項
3. `docs` 内の開発工程表と対象工程の実装指示書
4. `docs` 内の正式な仕様書・設計書
5. 既存コードの実装状態
6. 要確認事項としての報告

文書にない内容、または文書から一意に判断できない内容は、推測で確定しないでください。

---

## 9. 実装後の報告

実装後は、必ず次を報告してください。報告形式は `docs/ルールブック.md` §10 に従います。

- 判定
- 変更したファイル
- 変更した内容
- 変更していない内容
- 静的チェック結果
- `clasp push --force` の実行結果、または未実行理由
- `tasks/TASK_CURRENT.md` 更新内容
- 次の状態（確認待ちを明記）
- ユーザーへの依頼

---

## 10. 認証情報・秘密情報

次の中身を見ない・触らない・出力しないでください。

- 認証情報、Cookie、ブラウザプロファイル
- `.clasprc.json`
- APIキー（OpenAI APIキー等。GASスクリプトプロパティで管理される）
- トークン、その他の秘密情報

ユーザーにこれらを貼らせないでください。ユーザー共通の Codex 設定（環境のホームディレクトリ配下の `.codex` 等）には触れないでください。

---

## 11. 禁止事項（ユーザーの明示的な承認なしに行わない）

- ファイルの作成・編集・削除・移動・名前変更
- `appsscript.json` の変更
- clasp 操作 / デプロイ / 外部通信 / パッケージのインストール
- 仕様外の画面追加・機能追加、次工程の前倒し実装
- 既存データを破壊する処理
- 既存のシート名・列名・関数名・画面名の変更
- 正式文書にない仕様の確定（用語・列・関数・画面・条件の新造を含む）
- 文書間の矛盾を独自に解消すること
- ユーザー共通 Codex 設定への変更
