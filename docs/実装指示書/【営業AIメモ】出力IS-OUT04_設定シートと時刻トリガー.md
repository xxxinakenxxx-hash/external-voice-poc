## 3. 今回の作業

### 3-1. 設定シートの新規追加

スプレッドシートに設定用シートを1枚追加する（シート名の例：`output_config`）。次の運用パラメータを、キーと値の形で持つ。

| キー | 暫定値 | 意味 |
|---|---|---|
| `daily_report_enabled` | `TRUE` | 個人日報の通知配信をするか（オン/オフ） |
| `manager_summary_enabled` | `TRUE` | 日報まとめの通知配信をするか（オン/オフ） |
| `send_hour` | `18` | 配信する時刻（時。分単位が要る場合は `send_minute` を追加） |
| `collect_from_hour` | `0` | 集計対象の開始時刻（当日の何時から） |
| `collect_to_hour` | `18` | 集計対象の終了時刻（当日の何時まで） |
| `skip_weekend` | `TRUE` | 土日は配信しないか |

- **設定シートを読む関数**（例：`getOutputConfig()`）を作る。設定シートをキー・値で読み、オブジェクトで返す。
- 値が無い/シートが無い場合は、上表の暫定値をデフォルトとして使う（安全側）。

### 3-2. 配信の親関数（トリガーから呼ばれる）

- 例：`runDailyDelivery()`。時刻トリガーから毎日呼ばれる。
- 処理：
  1. `getOutputConfig()` で設定を読む。
  2. `skip_weekend` が TRUE で今日が土日なら、何もせず終了する。
  3. `daily_report_enabled` が TRUE なら、`notifyDailyReport()`（OUT-02）を呼ぶ。
  4. `manager_summary_enabled` が TRUE なら、`notifyManagerSummary()`（OUT-03）を呼ぶ。
  5. 例外は try-catch で捕捉し、`logError`（API-13）に記録する（配信失敗で全体が止まらないように）。

**注意：** 集計範囲（`collect_from_hour`〜`collect_to_hour`）は、OUT-02/03の集計処理が参照する。OUT-02/03が固定で「当日0〜現在」にしている場合、本工程で設定シートの値を見るように微修正する（該当箇所のみ）。

### 3-3. 時刻トリガーの登録

- **時刻トリガーの登録は、けんたろうがGASエディタのUIで行う**（コードからの `ScriptApp.newTrigger` 自動登録は、暫定では使わず、手動登録とする。誤って多重登録するのを防ぐため）。
- トリガーは「日付ベースのタイマー」で「午後6時〜7時」に `runDailyDelivery` を実行、で登録する。
- （GASの制約）時間主導トリガーの起動時間帯はトリガー登録で決まる。設定シートの `send_hour` は、トリガーが起動した後に「今が配信すべき時刻か」を判断する用途、および将来の分単位制御・記録用とする。暫定では、トリガーを18時台に登録しておけば、`runDailyDelivery` が18時台に走る。

## 10. けんたろうの実機確認手順

**手順1：** 設定シート（`output_config`）に、暫定値（第3-1章の表）が入っていることを確認する。

**手順2：** GASエディタで `runDailyDelivery` を手動実行する。営業スペースと管理職スペースの両方に通知が届くことを確認する（オン/オフが両方TRUEの状態で）。

**手順3：** 設定シートの `daily_report_enabled` を `FALSE` にして、再度 `runDailyDelivery` を手動実行する。**個人日報の通知が飛ばず、まとめだけ飛ぶ**ことを確認する（設定シートが効いている証拠）。確認後、`TRUE` に戻す。

**手順4：** GASエディタで、時刻トリガーを登録する。「トリガー」→「トリガーを追加」→ 実行する関数 `runDailyDelivery`、イベントのソース「時間主導型」、「日付ベースのタイマー」→「午後6時〜7時」。保存する。

**手順5：** **翌日の18時台**に、営業スペースと管理職スペースへ、実際に通知が自動で届くことを確認する（通し確認）。

**手順6：** 届いた通知のURLから、日報ページ・まとめページがそれぞれ開けることを確認する。

**手順7：** 結果を報告する。あわせて、**運用開始日**（今日または翌日の実配信日）を記録する。
