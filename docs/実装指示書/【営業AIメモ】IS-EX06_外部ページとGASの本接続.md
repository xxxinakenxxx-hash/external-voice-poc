## 2. 今回の作業

### 2-1. GAS側（`src/`）

**`submitMemo`（API-03）の改訂**

| 項目 | 変更前 | 変更後 |
|---|---|---|
| 呼び出し元 | S-02（google.script.run） | `doPost`（API-16） |
| 担当者の特定 | 関数内で `Session.getActiveUser().getEmail()` | **引数 `userKey`（doPostがトークン検証で確定したメールアドレス）を受け取る** |
| 引数 | `payload` | `payload`（`userKey` を追加。`inputText` は従来どおり） |

- **`submitMemo` 内の `Session.getActiveUser().getEmail()` 呼び出しを削除する。** 送信API（デプロイB）は匿名アクセスのため、ここでSession取得はできない。
- `lookupBranch(userKey)`（API-10）に、引数で受け取った `userKey` を渡す。
- 以降の処理（`callAiApi` → `parseAiResponse` → `writeRecordsToSheet`）は**既存のまま流用する。変更しない。**
- 戻り値（`success` / `recordCount` / `records` / `errorMessage`）は既存のまま。

**`doPost`（API-16）の改訂**

1. リクエストボディから `token` と `inputText` を取り出す。
2. `verifyToken(token)` を呼ぶ。**無効なら即座にエラーJSONを返して終了する（submitMemoを呼ばない）。**
3. 有効なら、`submitMemo({userKey: <確定したメールアドレス>, inputText: <本文>})` を呼ぶ。
4. 戻り値をJSONで返す。
5. 例外は try-catch で捕捉し、`logError`（API-13）に記録したうえで、エラーJSONを返す。

**CORS：** EX-02で成立した形式を維持する。Content-Type は `text/plain;charset=UTF-8`（プリフライトを避けるため）。**この形式を変更しない。**

### 2-2. 外部ページ側（`external/input-page/`）

**トークンの受領と保持**

1. ページ読み込み時に、URLのフラグメント（`#token=...`）からトークンを取り出す。
2. トークンを **sessionStorage** に保存する（localStorageではない。タブを閉じれば消える）。
3. **取り出した後、URLからフラグメントを消す**（`history.replaceState` 等）。URLバーにトークンが残らないようにする。
4. sessionStorageにトークンがない状態でページを開いた場合、「TOP画面からアクセスしてください」と表示し、送信ボタンを無効化する。

**送信処理**

1. 入力テキストが空でないことを検証する。
2. `unsent_text` をlocalStorageに保存する。
3. 「記録しています…」を表示し、送信ボタンを無効化する（二重送信の防止）。
4. GASの送信API（デプロイBのURL）へ `fetch` でPOSTする。送信するのは **`token` と `inputText` のみ**。
5. 応答を受け取る。
   - 成功時：結果を表示する（第2-3章）。**localStorageの `unsent_text` を削除する。**
   - 失敗時：エラーメッセージを表示する。**`unsent_text` は削除しない**（入力テキストを守る。要件定義FR-01・NFR-02）。
6. 送信ボタンを再度有効化する。

**送信先URL：** GASのデプロイBのURL。ソースに直書きせず、ページ内の設定用の定数として1箇所にまとめる（デプロイし直すたびにURLが変わるため、変更箇所を局所化する）。

### 2-3. 保存完了時の表示

**※ここは、けんたろうの承認内容に従うこと。**

- **案Aが承認された場合：** 「保存しました」＋分割された件数＋各レコードの顧客名・案件テーマ・宿題の有無を表示する（要件定義FR-07・8-5のまま）。
- **案Bが承認された場合：** 「保存しました」＋件数のみを表示する。

ChatGPTは、どちらが承認されたかを**明示してCodexに伝えること。** Codexが推測で決めてはならない。

## 3. やってはいけないこと

- `callAiApi`・`buildAiRequestPayload`・`parseAiResponse`・`writeRecordsToSheet`・`generateRecordId`・`markExtractionFailure`・`logError` を変更しない。**既存のまま流用する。**
- AIプロンプト（`getSystemPrompt`）を変更しない。
- `verifyToken` の検証ロジックを緩めない。
- トークンを **localStorage** に保存しない（sessionStorageを使う）。
- 商談本文をURLパラメータに載せない。
- 外部ページに担当者名・メールアドレス・営業所名・マスタ情報・過去の商談記録を表示しない。
- 外部JSライブラリ・CDN・解析タグを追加しない。
- 外部サーバー側に商談本文を保存しない。
- 位置情報・店舗検索（API-14/15・FR-09）を実装しない（IS-06Rで扱う）。
- Content-Type を `application/json` に変更しない（プリフライトが発生し、EX-02で成立した疎通が壊れる）。
- 正本文書を変更しない。

## 7. けんたろうへの作業依頼

**手順1：** GASのデプロイA（TOP用）とデプロイB（送信API用）を更新する（またはコードを反映した新しいデプロイを作る）。デプロイBのURLが変わった場合、外部ページ側の送信先URLの修正が必要になるため、Codexに連絡する。

**手順2：** TOP（デプロイA）を開く。担当者名が表示され、「入力画面へ」のリンクがあることを確認する。

**手順3：** 「入力画面へ」を押し、外部入力ページへ遷移する。**URLバーにトークン（`#token=...`）が残っていないこと**を確認する。

**手順4：** マイクボタンで音声入力し、商談内容を話す。テキストが入る。

**手順5：** 送信ボタンを押す。「記録しています…」の後、保存結果が表示されることを確認する。

**手順6：** スプレッドシートの `deal_records` を開き、**正しく記録されていること**を確認する。特に次を確認する。
- 担当者名（`user_name`）が自分の名前になっているか
- 営業所名（`branch_name`）が正しいか
- 商談内容が顧客×案件テーマ単位で分割されているか
- 抽出できなかった項目が赤背景になっているか

**手順7：** 送信成功後、ブラウザを閉じて外部ページを開き直す（TOPからでよい）。**前回の入力テキストが復元されないこと**（＝送信成功でlocalStorageが消えていること）を確認する。

**手順8：** TOPを経由せず、外部ページのURLを直接開く。**「TOP画面からアクセスしてください」と表示され、送信できないこと**を確認する。

**手順9：** 結果を報告する（手順2〜8の各結果、deal_recordsの記録内容）。

## 8. 完了条件

1. TOPからトークンを受け取り、外部ページのsessionStorageに保持する。URLバーにトークンが残らない。
2. 送信で `token` と `inputText` のみをPOSTする。
3. GASがトークン検証→担当者確定→lookupBranch→AI解析→分割→deal_records書き込みを行う。
4. 外部ページに結果が表示される（承認された案A／案Bに従う）。
5. 書き込み成功後、localStorageの `unsent_text` が削除される。
6. 失敗時は `unsent_text` が保持され、エラーが表示される。
7. error_log にエラーが記録される。
8. トークンなしで外部ページを直接開くと、送信できない。

## 9. 停止条件と戻り先

**停止条件：** 書き込みが行われない／担当者・営業所が誤って記録される／送信失敗時に入力テキストが失われる。

**戻り先：** 該当箇所（トークン検証／`submitMemo` の引数／書き込み／localStorage処理）を切り分けて修正する。EX-07へ進まない。

**特に重要：** 担当者が誤って記録される場合は、直ちに停止する。誰の商談記録か分からないデータが混入すると、後から修復できない。
