## 1. 前提

- EX-01が完了している。
- EX-02で、外部ページからGAS検証用 `doPost` へPOSTが到達し、応答を受け取れることが確認済みである。
- EX-02が停止条件に該当している場合、本工程は開始しない。
- 本工程は、GAS TOPでログイン中メールを取得できることと、外部ページから叩けるAPI公開方式が両立する見通しを確認するだけである。
- 本物のトークン、user_master照合、AI解析、スプレッドシート書き込みは実装しない。
- 本番 `doGet`、`validateUser`、`submitMemo`、`src/` 配下には触れない。

## 2. 今回の作業

1. GAS検証用プロジェクトに、ログイン中のメールアドレス取得結果だけを表示する最小の検証用 `doGet(e)` を新規作成する。
2. EX-02の検証用 `doPost(e)` と合わせ、次の構成を比較できる状態にする。
   - 同一デプロイでTOP認証と外部POSTを両立できるか。
   - デプロイ分離が必要か。
   - 別GASプロジェクトが必要か。
3. コードで確定できないデプロイ設定は変更せず、けんたろうの手動確認項目として整理する。
4. メール取得可否と外部POST到達可否を、構成ごとに記録できるREADMEまたは確認メモを追加する。

## 6. 作業後に必ず実行すること

- GAS検証用 `.gs` ファイルを一時 `.js` にコピーし、`node --check` を実行する。
- HTMLを追加した場合は `<script>` 部分を抽出して `node --check` を実行する。
- 一時ファイルは削除する。
- 表示する情報がメール取得結果だけであることを確認する。
- user_master、AI、スプレッドシートへ接続していないことを確認する。
- 本番ファイルを変更していないことを確認する。

## 7. 条件付き clasp push --force

- EX-02で使用した検証用GASプロジェクトが明確な場合のみ実行してよい。
- 検証用ファイル以外を変更していないことを確認してから実行する。
- 構文確認に失敗した場合は実行しない。
- 本番GASへ反映される可能性がある場合は実行しない。
- デプロイ設定変更はCodexが行わない。けんたろうが手動で行う。

## 10. ユーザー確認依頼

Codexは、けんたろうへ次を1つずつ依頼する。

1. 検証用GASのデプロイ設定を確認する。
2. TOP用URLを開き、ログイン中の自分のメールアドレスが表示されるか確認する。
3. 外部ページから検証用POSTを1回送信する。
4. POSTが到達し、外部ページに応答が表示されるか確認する。
5. 同一デプロイで両立したか、分離が必要だったかを報告する。

## 11. 停止条件

次のいずれかに該当した場合は停止する。

- GAS TOPでメールアドレスを取得できない。
- 外部POSTを受けられる公開設定とTOP認証が両立しない。
- デプロイ分離・別プロジェクト化でも成立の見通しが立たない。
- 本番コードを変更しなければ検証できない。

停止時は、トークン本実装、正本文書改訂、EX-04以降、IS-06へ進まない。
