## 3. 今回の作業

### 3-1. 本番ページ（external/input-page/）へのGPS・店舗選択UIの実装

1. **入力開始時（ページを開いた直後）にGPSを取得する。** ブラウザのGeolocation APIを使う。許可はブラウザ標準ダイアログ。GPS同意画面・ON/OFF切替UIは設けない。
2. GPS取得後、EX-08で作った送信API（デプロイB）へ `storeCandidates` アクションでPOSTする（token＋latitude＋longitude）。近い順4店舗を受け取る。
3. 受け取った4候補を、入力欄の近くに先に表示する。営業担当は、メモを音声・テキストで入力しながら、店舗をタップで選ぶ。
4. **該当店舗がない場合：** 店舗名の手入力欄に入力 → `storeSearch` アクションでPOST（token＋keyword）→ 名寄せ候補を表示 → 選択で確定。
5. **マスタにも該当なし：** 手入力した文字列をそのまま店舗名として使う。
6. **GPS取得失敗・拒否時：** 候補表示をスキップし、手入力に進む。**メモ入力と送信は妨げない**（フォールバックを必ず用意する）。
7. **送信は1回。** 入力と店舗選択を終えたら「送信」を1回押すだけで、メモ・選択した店舗・入力開始時に取得した緯度経度をまとめて `submit` アクションでPOSTする（token＋inputText＋latitude＋longitude＋storeName）。
8. GASが `submit` を処理し、AI解析・分割・書き込みに加え、deal_recordsのS・T・U列に緯度・経度・店舗名を記録する。

### 3-2. 正本文書の改訂

- **要件定義書 FR-09：** 「送信時にGPS取得」を「入力開始時（外部入力ページを開いた直後）にGPS取得」へ改訂する。取得した緯度・経度を訪問時の場所として deal_records のS〜U列に記録する旨、送信は1回で完結する旨を明記する。店舗確定ロジック（近い順4候補→選択／手入力→名寄せ→確定）は変更しない。

## 15. けんたろうの実機確認手順（iPhone Safari）

Codexの作業完了後、会社ケータイ（iPhone・Safari）で実施する。

**手順1：** TOP画面（デプロイA）を開き、ログインして「入力画面へ」で本番の外部入力ページへ遷移する。

**手順2（入力開始時GPS）：** ページを開いた直後に、GPSの使用許可ダイアログが出ることを確認する。許可する。

**手順3（候補先出し）：** メモを入力する前に、近い順の店舗候補（最大4件）が表示されることを確認する。

**手順4（入力と選択）：** マイクで商談内容を話しながら、店舗をタップで選ぶ。

**手順5（該当なし名寄せ）：** 候補に該当がない想定で、店舗名を手入力し、部分一致の名寄せ候補が出て、選べることを確認する。

**手順6（送信1回）：** 「送信」を1回押すだけで保存が完了することを確認する（店舗選択のために送信を複数回押す必要がないこと）。

**手順7（記録確認）：** deal_recordsを開き、S・T・U列に緯度・経度・店舗名が記録されていることを確認する。緯度・経度が、入力開始時（その店にいた時点）の場所であることを確認する。

**手順8（GPS失敗フォールバック）：** GPSを拒否（またはオフ）した状態でページを開く。候補が出ない代わりに、店舗名を手入力でき、メモ入力・送信が問題なくできることを確認する。

**手順9（既存機能の非破壊）：** 音声入力・テキスト入力・複数回の追記・通信断時の未送信復元が、これまでどおり動作することを確認する。

**手順10：** 結果を報告する（各手順の成否、S〜U列の記録内容、GPS失敗時の挙動）。
