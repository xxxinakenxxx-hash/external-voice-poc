## 2. 今回の作業

### 2-1. トークンの設計（この設計から逸脱しないこと）

**トークン形式：** `<payloadのbase64url>.<署名のbase64url>`

**payload（JSON）：**
```json
{ "uh": "<user_hash>", "exp": <有効期限のUNIX時刻（秒）> }
```

- `uh`（user_hash）：担当者のメールアドレスを、秘密鍵で HMAC-SHA256 したものを base64url にした文字列。
- `exp`：発行時刻 ＋ 12時間（43200秒）。
- **メールアドレス・氏名・営業所名をpayloadに含めてはならない。** トークンは外部ページが保持するため、中身は読まれる前提で設計する。

**署名：** payloadのbase64url文字列を、秘密鍵で HMAC-SHA256 し、base64url にしたもの。
GASでは `Utilities.computeHmacSha256Signature(data, key)` を使う。

**秘密鍵：** スクリプトプロパティ `TOKEN_SECRET` から取得する（`getAiApiKey`（API-11）と同じ方式）。**コードにハードコードしない。**

### 2-2. 実装する関数

| 関数 | 内容 |
|---|---|
| `getTokenSecret()`（API-19） | スクリプトプロパティ `TOKEN_SECRET` を取得する。未設定なら例外をスローする。 |
| `issueToken(email)`（API-17） | メールアドレスから `user_hash` を計算し、`exp`（現在時刻＋12時間）とともにpayloadを作り、署名してトークン文字列を返す。 |
| `verifyToken(token)`（API-18） | ①形式チェック ②署名を再計算して照合 ③`exp` が現在時刻を過ぎていないか確認 ④`uh` を user_master の全メールアドレスから計算した user_hash と突き合わせ、一致する担当者のメールアドレスを返す ⑤`active_flag` が「無効」なら拒否する。すべて満たせば `{valid: true, userKey: <メールアドレス>}`、それ以外は `{valid: false, errorMessage: ...}` を返す。 |
| `doGet(e)`（API-01・改訂） | TOP画面を返す。①`Session.getActiveUser().getEmail()` でメール取得 ②`validateUser()` で user_master 照合 ③有効なら `issueToken()` でトークンを発行し、担当者名と「入力画面へ」のリンクを表示する ④無効ならエラーメッセージを表示し、リンクを表示しない。 |
| `doPost(e)`（API-16・新規） | ①リクエストボディから `token` と `inputText` を取り出す ②`verifyToken(token)` を呼ぶ ③**無効なら即座に拒否して終了する（AI解析・書き込みに進まない）** ④有効なら、本工程では確定したメールアドレスをJSONで返して終了する（submitMemoへの接続はEX-06）。 |

### 2-3. TOP画面から外部ページへの遷移

- TOP画面に「入力画面へ」のリンク（またはボタン）を置く。
- リンク先は、外部入力ページのURL ＋ **フラグメント部にトークンを載せる**：`https://<外部ページURL>#token=<トークン>`
- **クエリパラメータ（`?token=`）ではなくフラグメント（`#token=`）を使う。** フラグメントはサーバーに送信されないため、外部ホスティングのアクセスログにトークンが残らない。
- 外部ページのURLは、スクリプトプロパティ `INPUT_PAGE_URL` から読む。ハードコードしない。

### 2-4. マニフェスト

- `src/appsscript.json` の `oauthScopes` に `https://www.googleapis.com/auth/userinfo.email` を追加する（EX-03で必要と判明したもの）。既存のスコープは消さずに追記する。

## 8. けんたろうへの作業依頼

Codexの作業完了後、順に実施する。

**手順1：スクリプトプロパティを設定する**
GASエディタ →「プロジェクトの設定」→「スクリプト プロパティ」で、次の2つを設定する。
- `TOKEN_SECRET`：長いランダム文字列（32文字以上。推測されない値。他人に見せない）
- `INPUT_PAGE_URL`：EX-04で公開した外部入力ページのURL

**手順2：デプロイBを作り直す（送信API用）**
「デプロイ」→「新しいデプロイ」→種類「ウェブアプリ」。実行するユーザー＝「自分」、アクセスできるユーザー＝「全員」。URLを控える。

**手順3：デプロイAを作り直す（TOP用）**
もう一度「新しいデプロイ」→種類「ウェブアプリ」。実行するユーザー＝「自分」、アクセスできるユーザー＝「社内ドメイン限定」。URLを控える。これがTOPのURLである。
※スコープを追加したため、既存デプロイの更新ではなく**新規作成**すること。承認画面が出たら承認する。

**手順4：TOPを開く**
手順3のURLを開く。自分の担当者名が表示され、「入力画面へ」のリンクが表示されることを確認する。

**手順5：トークンが発行されていることを確認する**
「入力画面へ」のリンクにカーソルを合わせる（またはリンク先URLを確認する）。URLの末尾に `#token=` に続く長い文字列が付いていることを確認する。**この文字列の中に、自分のメールアドレスがそのまま読める形で入っていないこと**を確認する。

**手順6：正しいトークンで送信APIを叩く**
手順5のトークン文字列をコピーし、EX-02の検証用外部ページ（`external/voice-poc/`）から、送信先URLに手順2のデプロイBのURLを入れ、トークンを貼って送信する。**自分のメールアドレスが返ってくること**を確認する。

**手順7：改ざんトークンが拒否されることを確認する**
手順6のトークンの文字を1文字だけ書き換えて送信する。**拒否されること**（メールアドレスが返らないこと）を確認する。

**手順8：トークンなしが拒否されることを確認する**
トークンを空にして送信する。**拒否されること**を確認する。

**手順9：結果を報告する**
手順4〜8の各結果を報告する。特に**手順7・手順8で確実に拒否されたか**を明記する。

## 9. 完了条件

1. TOP（デプロイA）でログイン中メールを取得し、user_master照合の上、有効担当者にトークンを発行する。
2. 無効・未登録の担当者にはトークンを発行せず、エラーメッセージを表示する。
3. doPost（デプロイB）が、正しいトークンを受け取ったときにメールアドレスを確定して返す。
4. **改ざんトークン・期限切れトークン・トークンなしを、すべて拒否する。**
5. トークンにメールアドレス・氏名・営業所名が含まれていない。

## 10. 停止条件と戻り先

**停止条件：** トークンの発行または検証が動作しない。または、改ざん・期限切れ・トークンなしを拒否できない。

**戻り先：** 署名方式（HMAC-SHA256）・秘密鍵の取得・base64urlのエンコード方法を切り分けて修正する。

**重要：** 拒否できない状態でEX-06へ進んではならない。送信APIは匿名公開であり、トークン検証が唯一の防御線である。ここが甘いまま本接続すると、誰でも他人になりすまして商談記録を書き込める状態になる。
