## 1. 工程名・目的

**工程名：** 配信土台（Chat Webhook送信）

**目的：** GASから、営業用・管理職用の2つのGoogle Chatスペースへ、Webhook経由でメッセージを投稿できることを確認する。日報・日報まとめの本体はまだ作らない。この工程では「GASからChatへ、狙ったスペースに、テキストを1本送れる」ことだけを確立する。

**なぜ最初にこれを作るか：** 配信の土台（Webhook送信）が動くことを先に確かめないと、日報・まとめ（OUT-02/03）で不具合が出たとき、送信の問題か中身の問題か切り分けられない。土台を単独で固める。

## 3. 今回の作業

### 3-1. 出力系のファイルを新規作成する

入力系（`src/` の 00〜05）と混ざらないよう、出力系のファイルを新規に作る。ファイル名は既存の採番規則に合わせる（想定：`src/06_output.gs`。実際の空き番号を確認して決める）。

### 3-2. 実装する関数

| 関数 | 内容 |
|---|---|
| `getChatWebhookUrl(target)`（API-22相当） | 引数 `target`（`'sales'` または `'manager'`）に応じて、スクリプトプロパティ `CHAT_WEBHOOK_SALES` / `CHAT_WEBHOOK_MANAGER` からWebhook URLを取得する。未設定なら例外をスローする |
| `sendChatMessage(target, text)`（API-21相当） | `getChatWebhookUrl(target)` でURLを取得し、`text` を Google Chat のメッセージ形式（`{ "text": text }`）でPOSTする。UrlFetchApp を使う。HTTPステータスと応答を返す |
| `testSendChat()`（検証用・末尾アンダースコアなし） | GASエディタから手動実行する検証関数。営業用と管理職用の両方へ、テストメッセージを1本ずつ送る。実行ログに送信結果（ステータス）を出す。**OUT-04完了後に削除する** |

### 3-3. 送信仕様

- Content-Type は `application/json; charset=UTF-8`。Google Chat Webhookは JSON を受け取る（入力側のdoPostのCORS回避とは別物。ここはGAS→外部の通常のPOSTで、CORS制約はない）。
- 送るのはテキストのみ（`{ "text": "..." }`）。この工程では、日報の中身ではなく「OUT-01 送信テスト」等の固定文言でよい。
- 送信先は、引数 `target` で営業用・管理職用を切り替える。

## 10. けんたろうの実機確認手順

第2章のチェックリスト（スペース作成・Webhook2本発行・スクリプトプロパティ設定）が済んでいる前提。

**手順2：** 実行ログに、営業用・管理職用の両方の送信結果（HTTPステータス）が出ることを確認する。両方 200 なら成功。

**手順4：** **管理職用スペース**を開く。同じくテストメッセージが投稿され、通知が来ることを確認する。

**手順5：** 結果を報告する（両スペースに届いたか、通知が来たか）。
