## 1. なぜ再検証するのか（前回の結果と原因）

初回のEX-03は、GASの公開設定を「自分として実行」「アクセスできるユーザー：全員」の1本のデプロイで検証した。この設定は匿名アクセスになるため、TOP画面でログイン中のメールアドレスが取得できない。一方で、この設定を社内ドメイン限定に変えると、今度は外部ページからのPOSTが通らない。

**1本のデプロイに「TOPのログイン必須」と「送信APIの匿名許可」の両方をやらせようとしたことが原因である。** 案B改の構成そのものが否定されたわけではない。

そこで、GASのデプロイを2本に分け、役割ごとに公開設定を変えて、両立するかを再検証する。GASは1つのプロジェクトに複数のデプロイを作成でき、公開設定はデプロイごとに個別に決められる。

---

## 2. 本工程の目的

GASのデプロイを2本に分けた構成で、次の2つが同時に成立することを確認する。

- TOP用デプロイ：Googleログイン中のメールアドレスが取得できる。
- 送信API用デプロイ：外部ページ（GitHub Pages）からのPOSTが到達し、応答が返る。

## 3. 検証する構成

| デプロイ | 用途 | 実行するユーザー | アクセスできるユーザー | 呼ぶ関数 |
| --- | --- | --- | --- | --- |
| デプロイA（TOP用） | ログイン確認・メール取得 | ユーザーとして実行（自分として実行でも可。§7で両方試す） | 社内ドメイン限定 | `doGet(e)` |
| デプロイB（送信API用） | 外部からのPOST受け口 | 自分として実行 | 全員 | `doPost(e)` |

コードは同じプロジェクト（`external/gas-poc/`）のまま。デプロイを2本作り、URLが2本に分かれる。TOPはデプロイAのURL、外部ページからの送信先はデプロイBのURLを使う。

## 5. Codexの作業内容

### 5-1. `external/gas-poc/Code.gs` の修正

`doGet(e)` を、TOP用デプロイの検証に必要な最小の情報を画面に表示する形にする。表示する内容は次の3つのみ。

1. `Session.getActiveUser().getEmail()` の取得結果（空文字なら「取得できず」と表示する）
2. `Session.getEffectiveUser().getEmail()` の取得結果（比較用。どちらが取れるかを切り分けるため）
3. 送信API用デプロイのURL（後述のスクリプトプロパティから読む。未設定なら「未設定」と表示する）

`doPost(e)` は EX-02 の検証用受け口のまま維持する。変更しない。

`doGet` で表示する送信API用URLは、スクリプトプロパティ `EX03_API_URL` から読む。ハードコードしない。値の設定はけんたろうが行う（§7）。

AI解析・スプレッドシート書き込み・user_master照合・トークン発行検証は実装しない。本工程の対象外である。

### 5-2. `external/voice-poc/index.html` の修正

EX-02で作った通信検証UIの送信先URLを、`external/gas-poc/` のどのデプロイURLに送るか、画面上のテキスト入力欄で差し替えられるようにする。デプロイBのURLを貼り付けて送信できるようにするためである。

送信するデータはEX-02と同じ固定値を維持する。

- `token=EX02_DUMMY_TOKEN`
- `text=EX-02 communication test`
- Content-Type：`text/plain;charset=UTF-8`（プリフライトを避けるため。変更しない）

音声入力部分は変更しない。

### 5-3. 静的チェック

- `external/gas-poc/Code.gs` を一時 `.js` に抽出して `node --check` を実行し、構文エラーがないことを確認する。一時ファイルは削除する。
- `external/voice-poc/index.html` の `<script>` 部分も同様に `node --check` で確認する。一時ファイルは削除する。

### 5-5. Git

## 7. けんたろうが行うこと（Codexの作業後）

順に実施する。

**手順1：デプロイBを作る（送信API用）**
GASエディタで「デプロイ」→「新しいデプロイ」→種類「ウェブアプリ」を選び、実行するユーザー＝「自分」、アクセスできるユーザー＝「全員」で作成する。表示されたURLを控える。

**手順2：スクリプトプロパティに送信API用URLを設定する**
GASエディタの「プロジェクトの設定」→「スクリプト プロパティ」で、`EX03_API_URL` に手順1のURLを設定する。

**手順3：デプロイAを作る（TOP用）**
もう一度「デプロイ」→「新しいデプロイ」を選び、実行するユーザー＝「ユーザーとして実行（ウェブアプリにアクセスしているユーザー）」、アクセスできるユーザー＝「社内ドメイン限定（丸菱のドメイン）」で作成する。表示されたURLを控える。これがTOP用のURLである。

**手順4：TOPを開いてメール取得を確認する**
手順3のURLをブラウザで開く。自分のメールアドレスが表示されるかを確認する。「取得できず」と出た場合は、実行するユーザーを「自分」に変えたデプロイをもう1本作り、同じく社内ドメイン限定にして開き、取得できるかを確認する（どちらの設定で取れるかを見極めるため）。

**手順5：外部ページから送信APIへPOSTする**
GitHub Pagesの外部ページを開き、送信先URLの欄に手順1のデプロイBのURLを貼り付けて送信する。HTTP 200が返り、`receivedToken` と `receivedText` が表示されるかを確認する。

**手順6：結果を記録する**
次の3点を報告する。

- TOP用デプロイでメールアドレスが取得できたか（取得できた場合、どの「実行するユーザー」設定だったか）
- 送信API用デプロイへの外部POSTが到達し、200が返ったか
- 上記2つが同時に成立しているか（TOP用デプロイの設定を変えたことで、送信APIが通らなくなっていないか）

## 8. 完了条件

次の3つをすべて満たすこと。

1. TOP用デプロイ（社内ドメイン限定）で、ログイン中のメールアドレスが取得・表示される。
2. 送信API用デプロイ（全員）へ、外部ページからPOSTが到達し、HTTP 200と応答本文が返る。
3. 上記1と2が、同一プロジェクトの2本のデプロイで同時に成立している。
