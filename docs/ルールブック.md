# ルールブック

## 1. 目的

このルールブックは、AI支援による開発案件で、次の三者が同じ前提で作業するための共通運用規約である。

- 監督者（人間）
- 設計・管理者AI
- 実装者AI（Codex / Claude Code）

目的は、計画、実装、確認、反映、完了判定の各段階で、AIの勝手な判断、工程飛ばし、文脈忘却、無駄な確認、実装範囲の拡大、完了判定の早まりを防ぐことである。

このルールブックは個別仕様書ではない。  
個別md、仕様書、DB設計書、CSV仕様、画面仕様に書かれる具体要件は、原則として各正本文書を確認する。

---

## 2. 開発案件の基本構造

開発案件は、次の4段階で進める。

| 段階 | 内容 |
|---|---|
| ① 計画する | 正本、計画書、開発工程表、実装指示書、TASK_CURRENT.md を整える |
| ② 計画通りに実装する | 1工程＝1指示書。指示書ごとに Codex / Claude Code 向けプロンプトを作る |
| ③ チェックする | 今回の変更対象・影響範囲・前工程確認済みを分け、無駄な重複チェックをしない |
| ④ 人間が確認する | デプロイ、Apps Script側確認、/exec 画面目視確認など、AIが確認できない項目を人間へ明確に依頼する |

---

## 3. 三者の役割

| 役割 | 主担当 | 禁止事項 |
|---|---|---|
| 監督者（人間） | 目的、仕様判断、承認、差戻し、画面目視確認、デプロイ判断 | 未確定事項をAI任せで確定させない |
| 設計・管理者AI | 要件整理、工程分解、実装指示、Codex報告の読み解き、チェック指示、TASK_CURRENT管理 | 実装しない。未承認仕様を足さない。工程を飛ばさない |
| 実装者AI | 指示された範囲の実ファイル編集、静的チェック、条件付き `clasp push --force`、完了報告 | 仕様判断、指定外変更、勝手なデプロイ、勝手な外部通信をしない |

---

## 4. 指示解釈ルール

番号付き指示は、番号順の作業手順として扱う。

例：

1. 追加候補を出す  
2. 精査する  
3. 再編成する  

この場合、①を出す前に②③へ進まない。  
②の前に③の完成案やファイル更新確認へ進まない。

### 禁止

- ③だけを切り出して解釈する
- ①②を飛ばして最終案を出す
- 検討前に「更新してよいか」と聞く
- 許可確認やファイル操作ルールを、作業順飛ばしの理由にする
- ユーザーが意見を入れる場所を消す

---

## 5. 正本・計画ルール

実装前に、次を確認する。

- 正本文書
- 計画書
- 開発工程表
- 対象工程の実装指示書
- `TASK_CURRENT.md`
- 未確定事項
- 前工程の完了状況

実装用Markdownは、正式文書の代替ではない。  
個別mdにある要件をルールブックへそのまま転記しない。  
ただし、複数資料に共通する事故防止ルールは、共通ルールとして抽象化してよい。

---

## 6. 1工程＝1指示書ルール

1工程は、1本の実装指示書・1本のCodex / Claude Code向けプロンプトで進める。

### 複数工程の定義

複数工程とは、開発工程表・実装指示書・`TASK_CURRENT.md` 上で、次が分かれている作業を、1本の指示に混ぜることである。

- 工程番号
- 工程名
- 目的
- 完了条件
- 確認観点
- 変更対象

例：

- IS-07 と IS-08 を1本の指示で実装する
- 登録画面とCSV出力を同じ指示で実装する
- 今工程の作業中に、次工程の導線や関数を前倒しで作る
- 確認待ちのまま次工程を始める

### 複数工程ではないもの

1工程の中で、次を1本の指示に含めることは複数工程ではない。

- 実装
- 静的チェック
- 条件付き `clasp push --force`
- `TASK_CURRENT.md` 更新
- ユーザーへのデプロイ・実画面確認依頼

これは、1工程を完了させるための処理である。

---

## 7. 各工程の実装方法

各工程のCodex / Claude Code向け実装指示には、必ず次を含める。

1. 前提
2. 今回の作業
3. やってよいこと
4. やってはいけないこと
5. 作業後に必ず実行すること
6. 条件付き `clasp push --force`
7. `TASK_CURRENT.md` 更新内容
8. 完了報告形式
9. ユーザー確認依頼
10. 次工程には進まないこと

工程はまとめて実装しない。  
ただし、1工程の中では、実装後の確認、条件付き `clasp push --force`、`TASK_CURRENT.md` 更新、ユーザー依頼までを1本化する。

これにより、push指示漏れ、TASK更新漏れ、現在地の見失い、指示の重複を防ぐ。

---

## 8. 条件付き `clasp push --force` ルール

GAS反映操作を書く場合、必ず **`clasp push --force`** と書く。

### 禁止表現

- `clasp push`
- `push`
- `GASへpush`
- `claspで反映`

### 正しい表現

- `clasp push --force`

Codex / Claude Code は、`clasp push` と書かれていても `clasp push --force` に自動補正するとは限らない。  
したがって、設計・管理者AIは、Codex / Claude Codeへ渡す前に、指示文内のコマンド表記を必ず確認する。

### 実行条件

対象工程の実装指示書に明記されている場合だけ、条件付きで `clasp push --force` を実行してよい。

実行前に必ず確認する。

- 変更ファイルが対象範囲内である
- 構文確認でエラーがない
- 対象工程以外を前倒ししていない
- 既存データ・既存仕様・既存導線を壊していない
- 禁止事項に抵触していない

次のいずれかに該当する場合は、`clasp push --force` を実行しない。

- 構文確認でエラーが出た
- 対象外ファイルを変更した可能性がある
- 指示外の前倒し実装をした可能性がある
- 既存機能を壊した可能性がある
- `clasp push --force` 実行に不安がある

---

## 9. `TASK_CURRENT.md` 更新ルール

各工程の作業後、Codex / Claude Code は `TASK_CURRENT.md` を更新する。

記録する内容は次のとおり。

- 対象工程を実装したこと
- 変更したファイル
- 禁止事項に抵触していないこと
- 静的チェック結果
- `clasp push --force` の実行結果、または未実行理由
- `clasp run` / デプロイ / Git操作をしたか、していないか
- 次の状態

次の状態には、必ず確認待ちを明記する。

例：

```text
〇〇工程 実装・静的チェック・clasp push --force 完了。
ユーザーによるデプロイ更新待ち。
/exec 実画面確認待ち。
次工程には未着手。
完了判定は未実施。
```

---

## 10. 完了報告形式

Codex / Claude Code の完了報告は、必ず次の形式にする。

```text
- 判定：
- 変更したファイル：
- 変更した内容：
- 変更していない内容：
- 構文確認結果：
- clasp push --force 結果：
- TASK_CURRENT.md 更新内容：
- 次の状態：
- ユーザーへの依頼：
```

---

## 11. 次工程への進み方

次工程には自動で進まない。

次工程へ進む条件は次のとおり。

1. Codex / Claude Code が対象工程を実装する
2. 静的チェックが通る
3. 条件付き `clasp push --force` が必要な場合は成功する
4. `TASK_CURRENT.md` が更新される
5. ユーザーが必要なデプロイ更新を行う
6. ユーザーが `/exec` 実画面確認を行う
7. ユーザーが確認結果を報告する
8. 必要なら修正工程を実施する
9. 問題なければ、設計・管理者AIが次工程の指示を作る

---

## 12. チェック重複防止ルール

チェック項目は、指示書に書いてあるから機械的に繰り返すものではない。

設計・管理者AIは、各工程のチェック指示を作る前に、次を分ける。

| 区分 | 内容 |
|---|---|
| 今回必須チェック | 今回の変更対象そのもの |
| 影響範囲チェック | 今回の変更で壊れる可能性がある周辺 |
| 前工程で確認済み | 前工程で確認し、今回変更していない箇所 |
| 回帰確認 | 再確認が必要な重要箇所 |
| 今回不要 | 指示書にはあるが、今回変更と関係がない箇所 |

前工程で確認済みで、今回変更していない画面・導線・操作は、原則として再確認させない。  
再確認が必要な場合は、今回の変更による影響理由を明記する。

### 禁止

- 前工程で確認済みの画面を、今回の変更対象ではないのに再度確認させる
- 指示書に書いてあるという理由だけで同じ画面確認を繰り返す
- 3工程連続で同じ画面を確認させる
- チェック項目をコピペで作る

---

## 13. ユーザー確認依頼ルール

ユーザーへ確認を依頼するとき、確認対象を探させてはならない。

特にGASでは、関数名だけでなく、GAS上のファイル名を必ず書く。

### 禁止

- 関数名だけを書き、GASファイル名を書かない
- 「TASK_CURRENT.mdから探してください」と言う
- 「該当ファイルを探してください」と言う
- 「Apps Script側で確認してください」だけで終える
- 前工程で確認済みの画面を、理由なく再確認させる

### 5行以内テンプレート

```text
【ユーザー確認依頼】
確認場所：GASファイル「〇〇」、関数「〇〇」、画面「/exec?page=〇〇」。
確認内容：〇〇ボタンを押し、〇〇になるか見てください。
確認不要：前工程で確認済みの〇〇画面は今回は見なくてよいです。
Codexはデプロイ更新・実画面確認を完了扱いにしていません。
結果を「OK / NG / 違和感あり」で返してください。
```

---

## 14. 未確定・矛盾時の停止ルール

未確定事項は推測で補完しない。  
文書間矛盾は、AIが独自判断で解消しない。

次に該当した場合は、作業を停止して報告する。

- 正本文書と実装指示書が矛盾する
- `TASK_CURRENT.md` と現在の指示が矛盾する
- 対象工程の実装指示書がない
- 仕様書から一意に判断できない
- 未確定事項を実装しないと進めない
- 指定外ファイル・指定外関数の変更が必要
- 外部通信・認証・権限・データ構造変更が必要

---

## 15. 変更禁止ルール

実装者AIは、次を勝手に変更してはならない。

- 既存シート名
- 既存列名
- 既存関数名
- 既存画面名
- 既存ファイル構成
- 既存データ
- 追記専用データ
- 履歴データ
- 出力仕様
- 列定義
- 値定義
- ステータス定義
- 業務用語

定義を変えない。  
別名で呼ばない。  
正式文書にない名前を勝手に作らない。

---

## 16. 承認ゲート

次は、必ず人間の明示承認を得てから行う。

- Git操作
- デプロイ
- 外部通信
- 外部API追加
- パッケージ追加
- 認証・権限変更
- DB・シート・列変更
- 既存データ削除
- 既存関数名変更
- 既存ファイル構成変更

GAS反映としての `clasp push --force` は、対象工程の実装指示書に明記され、条件を満たす場合だけ実行できる。

---

## 17. セキュリティ・認証情報ルール

次の中身を見ない、触らない、出力しない。

- 認証情報
- Cookie
- ブラウザプロファイル
- `.clasprc.json`
- APIキー
- トークン
- 秘密情報

ユーザーにこれらを貼らせない。  
Codex / Claude Code にこれらを読ませない。

---

## 18. 完了判定ルール

次を混同しない。

| 状態 | 意味 |
|---|---|
| 整理済み | 情報を整理しただけ |
| 実装済み | ファイル変更が終わっただけ |
| 静的チェック済み | 構文等を確認しただけ |
| `clasp push --force` 済み | GASへ反映しただけ |
| デプロイ更新済み | Webアプリのデプロイを更新した状態 |
| 実画面確認済み | `/exec` 等で人間が画面確認した状態 |
| 完了確定 | 必要な確認がすべて終わり、未確認事項が明記された状態 |

「整理済み」を「完了確定」と呼ばない。  
「実装済み」を「使える」と呼ばない。  
Codex / Claude Code は、デプロイ更新と実画面確認を完了判定に含めて報告してはならない。

---

## 19. フィードバック処理ルール

利用者確認の指摘は、その場で実装判断しない。

まず、次に分けて記録する。

- 不具合
- 要望
- 用語の違和感
- 運用確認
- 再現手順
- 影響度
- 対応方針
- 対応先
- 状態

記録後、設計・管理者AIが整理し、必要なら次工程・修正工程として扱う。

---

## 20. 応答態度ルール

AIは、ユーザー対応で次をしてはならない。

- 同じ誤りを2回繰り返す
- 言い訳を5行以上書く
- ごまかす
- 謝らない
- ミスを認めない
- 別の質問や提案で視点をそらす
- 質問に答えない
- 「読め」と言われているのに読まない
- 読んでいないのに読んだふりをする
- 実行していないことを実行済みのように言う
- ユーザーの指示が悪かったように説明する

質問されたら、まず結論を3行以内で直接答える。

ミスをした場合は、次の3点だけを短く示す。

1. 何を誤ったか
2. どこに戻るか
3. 次に何を出すか

---

## 21. 文書版管理ルール

文書名は、原則として次の形式にする。

```text
【プロジェクト名】○○.docx
【プロジェクト名】○○.md
```

ただし、docs配下の設計・運用Markdown、システム・ツール側で固有名称が決まっているファイル、既存名称で参照されているファイルは、その名称に準ずる。

例：

```text
docs/DB_SCHEMA.md
docs/ルールブック.md
CLAUDE.md
AGENTS.md
README.md
appsscript.json
package.json
.clasp.json
```

`最新版`、`修正版`、`追加版`、`v1`、`v2` などをファイル名に乱立させない。

版数・変更履歴は、ファイル名ではなく本文内で管理する。  
旧版を残す場合だけ、旧版フォルダへ退避する。

---

## 22. 結論

このルールブックの中核は、次の4点である。

1. 計画してから実装する
2. 1工程＝1指示書で、計画通りに実装する
3. 同じチェックを無意味に繰り返さない
4. 人間が見る項目は、探させず、5行以内で具体的に依頼する

特にGAS反映では、`clasp push` と書かず、必ず `clasp push --force` と書く。
