## B-2. 現在の処理経路

1. 本システムの利用者（sales／manager／sales_support／sysadmin）が「営業AIメモ｜利用開始・本人確認」を開く
2. GASが会社Googleアカウントの担当者を確認し、`user_master` の role を取得する
3. role が sales・manager・sysadmin の場合にのみ署名付きトークンを発行する。role が sales_support には発行しない
4. role が sales・manager・sysadmin の場合は「営業AIメモ｜商談入力」へのリンクと「日報を見る」の2ボタンを表示する。role が sales_support の場合は「日報を見る」の1ボタンのみを表示し、商談入力画面へのリンクは表示しない

6. 「営業AIメモ｜商談送信・保存API」へ送信する
7. GASがトークン検証、担当者確定、店舗の必須検証と確定、AI解析を行う
8. 「営業AIメモ｜商談記録データ」の`deal_records`へ保存する
9. 商談入力画面内に保存結果を表示する

# C. 確定事項

## C-1. 店舗指定

- 【確定事項】訪問商談、電話商談のどちらでも、商談先店舗の指定を必須とする
- 【確定事項】店舗マスタから選択した場合は、店舗IDを送る
- 【確定事項】店舗マスタに存在しない場合は、営業担当が入力した店舗名を送る
- 【確定事項】店舗IDと入力店舗名が両方とも空欄の場合は送信を拒否する
- 【確定事項】クライアント側だけでなく、GAS側でも必須条件を検証する
- 【確定事項】店舗を指定しない商談記録は作らない

## C-2. 商談区分

- 【確定事項】`input_method`を営業担当が画面で選択する
- 【確定事項】値は「訪問」「電話」の2値とする
- 【確定事項】既定値は「訪問」とする
- 【確定事項】GPS取得可否、店舗IDの有無、時間帯等から自動判定しない

## C-3. 訪問商談

- 【確定事項】訪問時は、商談先店舗と実際の訪問店舗を区別できる設計にする
- 【確定事項】`customer_store_id`は商談先店舗を示す
- 【確定事項】`visit_store_id`は実際に訪問した店舗を示す
- 【確定事項】マスタ店舗を指定した訪問では、両者へ同じ店舗IDを設定する
- 【確定事項】訪問時のみ、取得できた緯度・経度を保存対象とする

## C-4. 電話商談

- 【確定事項】電話商談でも商談先店舗の指定は必須とする
- 【確定事項】電話では`visit_store_id`を設定しない
- 【確定事項】電話では緯度・経度を保存しない
- 【確定事項】営業担当の現在地を商談先として扱わない
- 【確定事項】電話商談ではGPS候補を出さず、店舗名検索から電話相手を選ぶ

## C-5. 位置情報の取得タイミング

- 【確定事項】画面を開いただけでは位置情報を取得しない
- 【確定事項】訪問を選択した状態で「現在地から店舗を探す」を押したときだけ取得する
- 【確定事項】電話では位置情報を取得せず、店舗名検索だけを使用する
- 【確定事項】訪問と電話を切り替えた場合は、店舗選択と位置情報を破棄し、店舗を選び直す
- 【確定事項】位置情報を拒否・取得失敗した場合も、店舗検索または手入力による店舗指定を必須とする

## C-6. 顧客名の確定

- 【確定事項】顧客名は、営業担当が画面で指定した店舗を基にGASが確定する
- 【確定事項】店舗マスタから選択した場合、GASが`store_master`から正式名称を再取得する
- 【確定事項】クライアントから送られた店舗名文字列を、そのまま正式名称として信用しない
- 【確定事項】マスタ未登録店舗の場合は、営業担当が入力した名称を`customer_name_raw`として保持する
- 【確定事項】マスタ未登録の場合は、同じ値を`customer_name`にも格納する
- 【確定事項】マスタ登録済みかどうかは`customer_store_id`の有無で判別する
- 【確定事項】`customer_store_id`と`customer_name_raw`の両方が空欄の記録は作らない

## C-7. AIの役割とAI分割

- 【確定事項】AI出力から`customer_name`を削除する
- 【確定事項】AIは顧客店舗を決定しない
- 【確定事項】AIは案件テーマ、商談内容、宿題・ToDo、宿題期限、顧客業態の整理に限定する
- 【確定事項】1回の送信で指定する顧客店舗は1店舗とする
- 【確定事項】同一店舗内で、部門、相手、案件テーマ、商談内容が異なる場合にレコードを分割する
- 【確定事項】分割後の全レコードに、同じ指定店舗を設定する
- 【確定事項】音声中に別の店舗名、部門名、運営会社名、館内店舗名が出ても、指定店舗を変更しない
- 【確定事項】部門名・担当者名・館内店舗・運営会社との関係は`deal_content`へ残す
- 【確定事項】別顧客の商談を正式に記録する場合は、その店舗を選択して別送信する
- 【確定事項】商品名、品種、SKUが複数出ただけでは、機械的に別レコードへ分割しない

## C-8. `customer_type`（顧客業態）

- 【確定事項】`customer_type`は削除しない
- 【確定事項】顧客業態のAI推定情報として維持する
- 【確定事項】店舗マスタとの照合状態を表す項目としては使用しない

## C-9. 店舗照合状態の表し方

- 【確定事項】店舗照合状態を示す専用列は新設しない
- 【確定事項】`customer_store_id`あり＝マスタ照合済み、`customer_name_raw`あり＝マスタ未登録として既存項目から判断する
- 【確定事項】`input_method`は専用項目として持つ

## C-10. `extract_status`（抽出ステータス）

- 【確定事項】`extract_status`はAIによる分割、整理、抽出の成否を表す
- 【確定事項】判定対象は`deal_theme`と`deal_content`のみとする
- 【確定事項】顧客名、店舗マスタ登録有無、位置情報、商談区分、`customer_type`は判定対象に含めない
- 【確定事項】赤背景の対象も、空欄となった`deal_theme`または`deal_content`とする

## C-11. `deal_records`の列構成

- 【確定事項】既存A〜U列の物理位置は動かさない
- 【確定事項】既存列の途中へ新しい列を挿入しない
- 【確定事項】V列に`customer_store_id`を追加する
- 【確定事項】W列に`customer_name_raw`を追加する
- 【確定事項】X列に`visit_store_id`を追加する
- 【確定事項】Y列に`input_method`を追加する
- 【確定事項】Z列に`user_key`を追加する
- 【確定事項】U列`store_name_gps`は過去データ保全のため列を残す
- 【確定事項】U列への新規書き込みは停止する

## C-12. 記録者を識別する情報

- 【確定事項】Z列`user_key`へ会社メールアドレスを保存する
- 【確定事項】同姓同名、氏名変更、表記揺れの影響を受けない本人確認に使用する
- 【確定事項】入力画面、保存結果、個人日報、管理職向けまとめには表示しない
- 【確定事項】外部入力画面やAI APIへ送信しない
- 【確定事項】営業担当がスプレッドシート上で閲覧する項目にはしない
- 【確定事項】追記時の所有者確認に使用する

## C-13. 追記

- 【確定事項】追記画面は、会社GoogleアカウントでログインするGAS側の社内向け画面へ置く
- 【確定事項】外部入力画面には過去の商談記録を表示しない
- 【確定事項】終了した商談記録の元行を直接編集しない。ただし、sysadminによるAI抽出結果の品質是正（G列`deal_theme`・H列`deal_content`・L列`extract_status`）はこの禁止の対象外とする（C-21）
- 【確定事項】追記は別シートへ保存し、元の`record_id`へ紐づける
- 【確定事項】1件の商談記録へ複数の追記を保存できるようにする
- 【確定事項】最低限の項目は`addition_id`／`record_id`／`additional_record`／`added_at`／`added_by`
- 【確定事項】誰が、いつ、何を追記したかを必ず記録する
- 【確定事項】追記は商談件数、訪問件数、電話件数へ加算しない
- 【確定事項】AI分割後の複数レコードのうち、追記は対象の`record_id`にのみ紐づく
- 【確定事項】日報、週報、管理職向けまとめ、分析では、元記録と紐づく追記を一緒に参照できる設計とする

## C-14. 追記権限

- 【確定事項】salesは、自分の商談記録だけを閲覧・追記できる
- 【確定事項】managerは、自分を含む同じ`master_branch_name`（営業所）の担当者の記録を閲覧できる。他人の記録へ追記はできない
- 【確定事項】managerが複数いる営業所では、manager全員が同じ営業所の記録を閲覧できる
- 【確定事項】sales_support（営業事務・営業支援）は管理職向け日報まとめ（S-06）で全営業所・全営業担当の記録を閲覧できる。閲覧専用であり、追記はできない
- 【確定事項】ロールは sales（営業担当）／manager（営業管理職）／sales_support（営業事務・営業支援）／sysadmin（情シス管理者）の4値とする。admin_staffは現行ロールとして使用しない。sales_supportは管理者ではない
- 【確定事項】代理商談・代理電話コメント・代理追記は今回の対象外とする。代理コメント機能は次回検討事項として扱い、今回の実装仕様には含めない
- 【確定事項】sysadminは全記録を閲覧・追記できる

# E. 技術設計（確定）

## E-1. 商談区分の画面選択

- 【確定設計】`営業AIメモ｜商談入力`の入力欄より上に、「訪問／電話」の2択スイッチを常時表示する。初期選択は「訪問」。送信データに`inputMethod`（値：`訪問`／`電話`）を必須で含める。スイッチの選択そのものは、位置情報の取得を開始しない（取得の開始条件はE-2）。
- 理由：C-2で自動判定を禁止しているため、明示選択が唯一の取得手段。既定値「訪問」により、通常運用では操作が増えない。
- 影響する正本：要件定義書（新規FR）、画面一覧・項目定義、画面遷移図、案件概要書7-1
- 影響する画面：営業AIメモ｜商談入力
- 影響するAPI：API-16（doPostMemo）、API-03（submitMemo）
- 影響するDB：Y列`input_method`の新設
- 影響するAI：なし
- 影響する日報・管理職向けまとめ・分析：訪問件数・電話件数の集計が可能になる

## E-2. 店舗指定の画面動作と位置情報の取得タイミング

### （1）画面を開いた直後の状態

- 商談区分は「訪問」が選択されている
- 位置情報の取得は開始しない。許可ダイアログを出さない
- 店舗指定エリアには「現在地から店舗を探す」ボタンと店舗名の検索欄を表示する

### （2）位置情報を取得する唯一の契機

- 「訪問」が選択されている状態で、利用者が「現在地から店舗を探す」を押したときだけ取得を開始する
- 画面表示、区分の切替、音声入力の開始、送信ボタン押下では位置情報を取得しない

### （3）訪問を選択しているときの表示

| 状態 | 表示 |
|---|---|
| 初期 | 「現在地から店舗を探す」ボタン＋店舗名検索欄 |
| 位置情報の取得に成功 | 近い順に最大4件の店舗候補を表示。タップで確定 |
| 候補に該当なし | 店舗名検索欄で検索し、部分一致の候補から選ぶ |
| 検索でも該当なし | 入力した店舗名をそのまま店舗指定として確定する（マスタ未登録扱い） |
| 位置情報を拒否された／取得に失敗した | 候補を出さず、「現在地を取得できませんでした。店舗名で検索してください」と表示し、検索欄へ誘導する。送信自体は止めない。店舗指定は必須のまま |

### （4）電話を選択しているときの表示

- 「現在地から店舗を探す」ボタンを表示しない
- 店舗名検索欄のみを表示する
- 検索で該当がない場合は、入力した店舗名をそのまま店舗指定として確定する
- 位置情報を取得済みであっても、送信データに緯度・経度を含めない

### （5）区分を切り替えたときの店舗選択状態

| 切替方向 | 店舗選択 | 位置情報 | 表示 |
|---|---|---|---|
| 訪問 → 電話 | 破棄する（未選択に戻す） | 破棄する（送信データにも含めない） | 店舗候補リストを閉じ、検索欄のみにする |
| 電話 → 訪問 | 破棄する（未選択に戻す） | 保持しない。再取得もしない | 「現在地から店舗を探す」ボタンを再表示する。押されるまで取得しない |

切替時には「商談区分を変更したため、店舗を選び直してください」と画面に表示する。

### （6）送信データに緯度・経度を含める条件

- 商談区分が「訪問」であり、かつ「現在地から店舗を探す」で位置情報を取得できていた場合のみ

- 理由：位置情報の許可を求めるのは、利用者が「現在地から探す」と自ら操作したときだけにすることで、電話商談の利用者に不要な許可ダイアログを出さずに済む。一度「許可しない」を選ばれると、以後その端末で位置情報が使いにくくなるため、無用な許可要求は避ける。訪問商談ではボタンを1回押すだけで近隣候補が出る。
- 実装済み画面への影響：外部入力画面の実装工程では、位置情報を「ページを開いた時点で取得する」方式を採用済みである。本設計はこれを変更するため、実装済みの入力画面に修正が発生する。
- 影響する正本：要件定義書FR-09、画面一覧・項目定義、画面遷移図、技術選定理由書3-9、案件概要書7-2
- 影響する画面：営業AIメモ｜商談入力
- 影響するAPI：API-14（getStoreCandidates。訪問かつボタン押下時のみ）、API-15（searchStoresByName。訪問・電話とも使用）
- 影響するDB：S列・T列は訪問時のみ値が入る
- 影響するAI：なし
- 影響する日報等：位置情報を伴う訪問実績の集計が可能になる

## E-3. 送信データとサーバー側の必須検証

送信データを次の項目で構成する。

| 項目 | 型 | 必須 | 内容 |
|---|---|---|---|
| token | 文字列 | 必須 | 本人確認用の署名付きトークン |
| inputText | 文字列 | 必須 | 商談テキスト本文 |
| inputMethod | 文字列 | 必須 | `訪問`／`電話` |
| customerStoreId | 文字列 | 条件付き | マスタから選んだ場合の店舗ID |
| customerNameRaw | 文字列 | 条件付き | 手入力した店舗名 |
| latitude / longitude | 数値 | 任意 | 訪問かつ位置情報を取得できた場合のみ |

GAS側（API-16）で、トークン検証の直後に次を検証する。該当した場合はAI解析へ進まず拒否する。

1. `inputMethod`が`訪問`／`電話`以外、または未指定
2. `customerStoreId`と`customerNameRaw`が両方とも空（前後の空白を除去した後で判定）
3. `customerStoreId`が`store_master`に存在しない
4. `customerStoreId`に対応する店舗の`active_flag`が「無効」
5. `inputMethod`＝`電話`で緯度・経度が届いた場合は、拒否せず緯度・経度だけ破棄して処理を続行する

## E-10. 追記の論理設計

- 元の商談行は一切更新しない。`deal_content`・`created_at`は保持する
- 追記は独立したデータとして保存し、`record_id`で元記録に紐づける
- 1件の`record_id`に複数の追記を紐づけられる
- 追記は商談件数・訪問件数・電話件数に加算しない
- 日報（S-05）・週報・管理職向けまとめ（S-06）・分析では、`record_id`をキーに元記録と追記を一緒に読む
- 元記録の直下に「追記（日付・追記者）」として表示し、元記録と追記の関係が分かる形にする
- 複数の追記がある場合は、追記日時順に表示する
- 店舗名の誤りも自由文の追記として残し、V列`customer_store_id`は書き換えない

- 理由：C-13・C-15・C-16の直訳。
- 影響する正本：DB設計書、API一覧、画面一覧・権限表、要件定義書、画面遷移図
- 影響する画面：追記画面（S-04）、個人日報ページ（S-05）、管理職向け日報まとめページ（S-06）
- 影響するAPI：追記系2関数
- 影響するDB：新規シート
- 影響するAI：日報・週報生成時に追記を併読する
- 影響する日報等：件数には加算せず、内容には反映する

## E-11. AI出力キーから`customer_name`を削除する

AIの出力キーを、`customer_name`を除いた5つ（`customer_type`／`deal_theme`／`deal_content`／`todo_item`／`todo_deadline`）とする。顧客名は、営業担当が画面で指定した店舗を基にGASが確定する。AIは顧客店舗を決定しない。

出力はトップレベルのJSON配列とする。`response_format`は指定しない。5キーのトップレベルJSON配列のみを返すことは、System Promptの指示で担保する。API事業者はOpenAI APIで確定とし、具体的なモデル名は正本で固定せず、実装上の変更可能な設定値として扱う（C-24）。

- 理由：キーを残したまま「保存には使わない」とすると、正本上に「AIが顧客名を抽出する」という旧経路の記述が残り、今回の是正が完結しない。
- 影響する正本：API一覧（API-06、外部API呼び出し仕様の出力フォーマット）、DB設計書2-1 E列、要件定義書FR-04
- 影響する画面：保存結果表示（顧客名が確定値になる）
- 影響するAPI：API-06のマッピングから顧客名を除去
- 影響するDB：E列の取得元が変わる（列位置・列名は不変）
- 影響するAI：AIプロンプト設計書§2・§3・§8の出力キー定義、§9のfew-shot例4件すべて、§11の検証観点
- 影響する日報等：顧客名が空欄になる記録が発生しなくなる

## E-15. 追記機能の物理設計

### （1）配置と認証

- 追記画面（S-04）は、会社Googleアカウントでログインする`営業AIメモ｜利用開始・本人確認`（デプロイA・社内ドメイン限定）へ置く
- TOP画面（S-00）に表示するボタンは role により異なる。sales・manager・sysadmin は「商談メモを入力する」「日報を見る」の2ボタン、sales_support は「日報を見る」の1ボタンのみとする。いずれの role でも追記専用ボタンを追加しない
- 「日報を見る」の遷移先は role により異なる。sales・sysadmin は個人日報（S-05）、manager・sales_support は管理職向け日報まとめ（S-06）を開く
- 追記画面（S-04）へ進めるのは、S-05を利用する sales・sysadmin のみである。S-05に表示された対象の商談記録を選択して追記する。追記対象は`record_id`で特定する
- TOP画面から追記画面へ直接遷移する導線は設けない
- 外部入力画面（S-02）から追記機能への導線は設けない。過去の商談記録を外部入力画面へ出さない
- 新しい利用者向け入口URL、新規GASデプロイ、新規外部ページは追加しない
- 元記録の所有者確認をサーバー側で行う（判定方法はE-16）
- 追記は別シートへ保存し、元行を更新しない（sysadminによる品質是正はC-21のとおり別扱い）

### （4）権限とサーバー側の認可

| ロール | 一覧の閲覧範囲 | 追記 |
|---|---|---|
| sales（営業担当） | 自分の記録のみ | 自分の記録のみ可 |
| manager | 自分を含む同じ`master_branch_name`（営業所）の担当者の記録 | 不可 |
| sales_support（営業事務・営業支援） | S-06で全営業所・全営業担当の記録（全件）。閲覧専用 | 不可 |
| sysadmin | 全記録 | 全記録へ可 |

閲覧できるかどうかと追記できるかどうかは別に返す。一覧取得API（API-36）の戻り値に`canAdd`を持たせ、追記できる記録だけ追記ボタンを表示する。`canAdd`の値にかかわらず、保存時はサーバー側（API-37）で必ず権限を検証する。

サーバー側で、ログイン中のメールアドレスと元記録のZ列`user_key`を照合する。一致しない場合は、`user_master.role`が`sysadmin`のときのみ許可し、それ以外（manager・sales_supportを含む）は拒否する。画面側の制御だけに依存しない。`sales_support`は自分の記録を含め一切追記できない。

### （5）追記後の表示と戻り先

- 追記保存後は、遷移元の個人日報（S-05）の該当記録へ戻り、追記一覧に追記日時順で反映する

- 理由：追記に音声入力は不要であり、入力画面を外部化した理由（マイク権限）が当てはまらない。外部入力画面に置くと、過去の商談記録を外部ページへ出すことになり、NFR-03「外部入力画面には過去の商談記録を一切出さない」に抵触する。また匿名公開の送信APIに読み取り処理を追加することになり、外部からの攻撃対象が広がる。GAS側なら本人確認がログイン情報だけで完結し、出力側（日報ページ・管理職向けまとめページ）と同じ認証方式に揃う。

## E-16. 記録者を識別するZ列`user_key`

### （1）表示名では所有者確認ができない理由

`deal_records`は、記録者をD列`user_name`（担当者フルネーム＝表示名）でしか持っていない。表示名で所有者を判定すると、次の問題が起きる。

| 問題 | 内容 |
|---|---|
| 同姓同名 | 全拠点50名へ展開したときに同姓同名が発生すると、他人の商談記録へ追記できてしまう |
| 氏名の変更 | 担当者マスタの氏名を更新すると、過去の記録に残る旧姓と一致しなくなり、本人が自分の記録へ追記できなくなる |
| 表記の揺れ | 姓と名の間が全角空白・半角空白・空白なしのいずれかで登録された場合、照合が外れる |

### （2）会社メールアドレスを使う理由

会社のGoogleアカウントのメールアドレスは、担当者マスタで重複を許さない唯一のキーとして確定している（DB設計書2-2 A列）。上記3つの問題をいずれも受けない。ログイン時に必ず取得できるため、追加の入力も不要である。

### （3）代替キーを検討した結果

| 候補 | 判定 |
|---|---|
| A列`record_id` | 記録そのものの識別子であり、誰が記録したかを表さない。代替不可 |
| D列`user_name` | 一意性がない。上記（1）の3問題を受ける。代替不可 |
| 社員番号 | 担当者マスタに列が存在しない。新設すると担当者マスタと商談記録の両方に列を追加することになり、採番規則も新たに決める必要がある。代替不可 |
| 署名付きトークンで使う変換済み文字列 | 署名用の秘密鍵を変更すると、過去に保存したすべての値と照合できなくなる。恒久保存するデータには適さない。代替不可 |

会社メールアドレス以外に代替できる既存キーは存在しない。

## E-17. エラー条件とエラーメッセージ

GAS側（API-16）でトークン検証の直後に判定し、該当したらAI解析へ進まずに拒否する。

| 条件 | 処理 | 画面に表示する文章 |
|---|---|---|
| 店舗ID・手入力店舗名がともに空 | 送信しない | 商談先の店舗を指定してください。「現在地から店舗を探す」を押すか、店舗名で検索してください。 |
| 店舗名が空白文字だけ | 店舗未指定として扱い、送信しない | 同上 |
| 存在しない店舗ID | 送信しない | 選んだ店舗が見つかりませんでした。もう一度、店舗を選び直してください。 |
| 無効店舗（`active_flag`＝無効） | 送信しない | この店舗は現在使用できません。管理者に連絡してください。 |
| 商談区分が不正または未指定 | 送信しない | 訪問か電話かを選んでください。 |
| 電話商談で緯度・経度が届いた | 緯度・経度だけ破棄して処理を続行 | 表示しない |
| 位置情報を拒否された／取得に失敗した | エラーとしない。店舗検索へ誘導する | 現在地を取得できませんでした。店舗名で検索してください。 |
| AI APIのタイムアウト | 商談記録を保存しない。入力テキストを保持する | 現在利用できません。しばらく経ってから、もう一度送信してください。 |
| AI APIのHTTP 4xx（認証エラー・リクエスト不正等） | 商談記録を保存しない。入力テキストを保持する。`error_log`へ記録する | 現在利用できません。管理者に連絡してください。入力内容は保持しています。 |
| AI APIのHTTP 5xx（サーバー側の一時障害） | 商談記録を保存しない。入力テキストを保持する | 現在利用できません。しばらく経ってから、もう一度送信してください。 |
| AIから応答は得たがJSON解析不能 | 元の文章全体を1レコードとして保存し、`extract_status`を「要確認」とする。確定した店舗情報は正常に保存する | 保存しました。内容の整理がうまくいかなかったため、後で管理者が確認します。 |
| スプレッドシートへの書込失敗 | 保存失敗として扱う。入力テキストを保持する | 保存に失敗しました。しばらく経ってから、もう一度送信してください。 |

- 理由：営業担当が次に何をすればよいかが分かる文章に統一し、システム内部の理由は出さない。空白文字だけの店舗名を店舗未指定と同じ扱いにするのは、営業担当から見て両者の区別がつかないため。電話商談で緯度・経度が届いた場合をエラーにしないのは、画面で訪問から電話へ切り替えた際に値が残りうるためで、これをエラーにすると送信できない状態に陥る。JSON解析不能のときだけ保存するのは、AIから応答が返っている以上、元の文章を失わずに残せるためである。タイムアウト・HTTPエラーではAIから何も返っておらず、この時点で保存すると店舗情報だけの空レコードが残る。未送信テキストを保持して再送を案内するほうが、記録の欠損も重複も起きない。
- 影響する正本：API一覧API-16のエラーハンドリング表、要件定義書FR-08・9-2、画面遷移図のエラー分岐、画面一覧・項目定義
- 影響する画面：営業AIメモ｜商談入力
- 影響するAPI：API-16、API-03
- 影響するDB：なし
- 影響するAI：検証をAI呼び出し前に行うため、無効な送信でAPI料金が発生しない
- 影響する日報等：店舗未指定レコードが発生しなくなる

## E-18. 店舗マスタの運用設計

### （1）登録担当

- 会社が指定した店舗マスタ管理担当者が登録する。個人名で担当者を固定しない
- 営業担当には店舗マスタを直接登録させない
- 必要作業は、手入力店舗の確認、重複確認、正式名称・住所・緯度経度の登録とする

### （2）整備頻度

- 週1回確認する。既存の週次確認（抽出失敗行の確認）と同じ日に実施する
- 店舗マスタ登録は、週次作業に追加される作業として運用要件へ明記する
- 手入力店舗が0件の週は登録作業を行わない

### （3）登録単位

- 実際の請求先・取引先の単位で登録する
- ホテル一括取引なら「ホテル1件」として登録する
- 各店舗との個別取引なら「店舗ごと」に登録する
- 部門名・売場名だけの違いは、店舗を分けず商談内容へ残す
- 顧客別集計も、この取引単位で行う

### （4）store_masterへの列追加

登録作業に住所が含まれるため、`store_master`に住所列を持たせる。既存A〜E列（`store_id`／`store_name`／`latitude`／`longitude`／`active_flag`）の位置は動かさず、F列に`address`を追加する。

- 理由：緯度・経度の登録が必要なため、営業担当がその場で登録するのは現実的でない。取引単位で登録することで、顧客別集計の単位が請求・取引の実態と一致する。
