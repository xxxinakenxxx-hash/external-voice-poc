## 0. なぜ今これを作るか

入力（貯める）はEX-09まで完了した。しかし出力（活用）がゼロだと、営業が入力しても誰も読まず、入力する動機が生まれず、システムごと形骸化する恐れがある。最低限の出力として、日報と日報まとめを先に立てる。

朝のTODO配信は、宿題が溜まって期限が動き出してから意味を持つため、立ち上げ初期は対象外とする（データが育ってから別途）。日報は、当日入力が1件でもあれば中身が出るため、立ち上げ初期から機能する。これを出力の第一歩とする。

## 1. 確定した方針（けんたろうと合意済み）

### 1-1. 届け方の基本思想

**通知（push）は「できたよ・ここから見て」の合図だけ。商談の中身はpushに載せない。中身は画面（ログイン必須）を開いて見る。**

これにより、Google Chatに商談内容が残らない。中身はスプレッドシートと画面（ログイン必須）の中だけにあり、Chatには「できました＋URL」しか流れない。入力側で貫いてきた「外部に中身を出さない」思想と一貫する。

### 1-2. 日報（営業個人向け）

- 営業6人が入るChatスペースへ、毎日18時に「日報ができました→(共通URL)」の**通知pushのみ**（中身なし）。
- 本体は、**日報専用の別ページ**で表示する。TOP画面に「日報を見る」ボタンを置き、押すとこの別ページへ遷移する。
- 別ページは、**ログイン者本人の日報を、直近30日ぶん、日付ごとに新しい順で並べる**（当日分が一番上）。開けば当日分がまず目に入り、下にスクロールすれば近い日も見返せる。
- 他人の日報は見えない。URLは全員共通でよい（誰が開くかはログインしたメールアドレスで決まるため）。
- 「各自の日報を全員に流すのは気が引ける」という懸念を、構造的に解消する（中身を流さないため）。

**なぜ別ページにするか：** Phase 2で検索機能（日別・店舗別・内容別）を、この同じページに足す前提。器を最初から独立ページにしておけば、検索追加時に作り直しにならない。

### 1-3. 日報まとめ（管理職向け）

- 管理職が入るChatスペースへ、毎日18時に「日報まとめができました→(共通URL)」の**通知pushのみ**（中身なし）。
- 本体は、**まとめ専用の別ページ（HTML）**で、6人分を表にして並べる。管理職（またはけんたろうが指定した人）だけが開ける。
- 表の列は「担当者（誰が）／訪問先（どこに）／商談要点（何を話したか）／日付」。1商談＝1行。
- 詳細（全文）は載せない。気になった行は、管理職がスプレッドシートの該当レコードを見るか、本人に聞く。

### 1-4. 配信手段

- Google Chat の受信Webhook。会社の受信Webhookは**許可済み（情シス確認済み・依頼不要）**。けんたろうがスペース設定からWebhook URLを発行できることを実機で確認済み。
- Webhookは**2本**。営業スペース用と、管理職スペース用。
- Webhook URLは秘密情報として扱う（漏れると誰でも投稿できる）。GASのスクリプトプロパティで管理し、コードに直書きしない。TOP画面のTOKEN_SECRETと同じ扱い。

### 1-5. 配信時刻と設定シート

- 配信時刻は**毎日18時**。
- 時刻・オン/オフ・対象範囲などの運用パラメータは、**設定用シートを1枚追加**して、そこで変更できるようにする。GASは配信時にこのシートを読んで動く。
- （GASの仕組み上の注意）時間主導トリガーの起動時間帯そのものはトリガー登録で決まる。トリガーは18時台に起動するよう登録し、分単位の時刻・オン/オフ・範囲は設定シートで制御する、という二段構成とする。

### 1-6. スコープ

- 対象：日報（個人）・日報まとめ（管理職）の2つ。来週同時リリース。
- 対象外：朝のTODO配信（データが育ってから別途）。停滞検知・ブリーフィング・見積ルーティング等のPhase 2活用群。
- 個人日報の検索機能（日別・店舗別・内容別）は**Phase 2**（第7章のデッドラインに注意）。
- 暫定・修正前提。

## 2. 全体の流れ

### 2-1. 日報（営業個人）

```
毎日18時の時刻トリガー（GAS）＋ 設定シートの値を読む
　↓
deal_records から「当日（0時〜18時）分」を担当者ごとに集計
　↓
営業スペースへ通知push（Webhook・営業用）
　「日報ができました。ここから確認 → https://<日報ページURL>」
　↓
営業担当がURLを開く → 日報専用ページ → Googleログインで本人特定
　↓
ログイン者本人の直近30日ぶんの日報を、新しい順で表示（当日が一番上・他人の分は出さない）
```

### 2-2. 日報まとめ（管理職）

```
毎日18時の時刻トリガー（GAS）＋ 設定シートの値を読む
　↓
deal_records から「当日（0時〜18時）分」を全担当者ぶん集計
　↓
管理職スペースへ通知push（Webhook・管理職用）
　「日報まとめができました。ここから確認 → https://<まとめページURL>」
　↓
管理職がURLを開く → まとめ専用ページ → ログインで本人特定 → role が管理職か確認
　↓
6人分（当日分）を表で表示（担当者／訪問先／商談要点／日付）。管理職以外は開けない
```

**集計範囲：** 当日0時〜配信時点（18時）まで。18時以降に入力された分は、その日の配信には載らない（翌日にも回らない）。この割り切りで運用する。

## 3. データソース（既存のdeal_recordsを読むだけ）

日報・日報まとめに必要な情報は、すべてdeal_recordsに既にある。新しい列は作らない。

| 表示項目 | deal_records列 | 備考 |
|---|---|---|
| 誰が（担当者） | D: user_name | まとめ表の1列目 |
| どこに（訪問先） | E: customer_name ／ U: store_name_gps | 顧客名を基本。GPS店舗名の併記は後で（暫定は customer_name） |
| 何を話したか（商談要点） | G: deal_theme ＋ H: deal_content | 要点として短く。全文は載せない |
| 日付 | B: created_at | 当日抽出・直近30日抽出のキー |
| （個人日報の本人特定） | D: user_name ／ user_master.user_key | ログインメールから担当者を特定して絞り込む |
| （詳細への導線） | K: original_text | 表・一覧には出さない。スプレッドシートで見る |

**書き込みは一切しない。読むだけ。** deal_recordsの構造は変更しない。

**保存と表示の分離：** deal_records は消さない（保存は無期限）。個人日報ページの初期表示だけを直近30日に絞る。30日より古い記録もデータとしては残り、Phase 2の検索で到達できる。

## 4. 認証・本人特定（既存の仕組みを流用）

### 4-1. 個人日報ページ

- Googleログイン・本人特定（Session.getActiveUser().getEmail()）は、既存のTOP画面の仕組みをそのまま使う。
- ログインしたメールアドレスで user_master を引き、その担当者の直近30日分だけを deal_records から絞って表示する。
- 他人の分は取得しない（クエリの段階で本人の user_name / user_key で絞る）。

### 4-2. 管理職まとめページ

- まとめページも、ログインで本人を特定する。
- **user_master の role 列（sales／admin_staff）を使い、管理職（またはけんたろうが指定するrole）以外は開けないようにする。**
- role列は既にDB設計書に定義済み。新しい仕組みを作らず、この列の値で判定する。
- 誰を「管理職」とみなすか（role=admin_staff か、専用のrole値を足すか、けんたろう自身を含めるか）は第7章で確認。

## 5. 対応API（正本との関係・暫定スコープ）

正本のAPI一覧には、配信・日報系のAPIが既に定義済み。本工程は、そのうち日報・日報まとめに必要な部分だけを暫定実装する。

| 正本API | 本工程での扱い |
|---|---|
| API-21 sendChatMessage（Google Chat Webhookへ送信） | **使う。** 暫定は「通知（URL）のみ送る」用途に限定 |
| API-22 getChatWebhookUrl（スクリプトプロパティからWebhook URL取得） | **使う。** 営業用・管理職用の2本を管理 |
| API-28 generateManagerReport（全レコード集計→日報/週報生成） | **一部使う。** 暫定は「集計して表・一覧に並べる」だけ。AI要約の作り込みはしない（deal_theme＋deal_contentをそのまま要点として使う） |
| push_log（配信履歴シート） | **暫定は任意。** 配信の記録を残すなら使う。最小では省略可（第7章で確認） |
| API-20 runMorningBriefing／API-24 scanTodosForReminder（朝TODO系） | **使わない。** 朝TODOは対象外 |

**暫定の肝：** API-28は本来AIで日報文を生成する重い設計だが、暫定では「deal_recordsの当日分を、担当者・訪問先・要点の表／一覧に並べる」だけにする。AI要約は後から足せる（修正前提）。

## 6. 工程分解（来週同時リリース）

配信の土台を先に確かめてから、日報・日報まとめの中身を乗せる。最後に設定シートと時刻トリガーで自動化する。

| 工程 | 名称 | 目的 | 前提 | GAS反映 | けんたろう確認 |
|---|---|---|---|---|---|
| OUT-01 | 配信土台（Chat Webhook送信） | GASから営業用・管理職用の2スペースへ、Webhook経由でテスト投稿できることを確認する | Webhook URL 2本を発行済み | あり | あり（2スペースにテスト投稿が届く） |
| OUT-02 | 個人日報ページ（別ページ・直近30日） | 日報専用の別ページを作り、ログイン者本人の直近30日分を新しい順で表示。TOP画面に「日報を見る」ボタンを置く | OUT-01完了 | あり | あり（本人の分だけ・当日が上・数日分見返せる） |
| OUT-03 | 日報まとめページ（別ページ・6人表） | まとめ専用の別ページを作り、当日分の6人を表で表示。管理職だけが開ける | OUT-02完了 | あり | あり（6人分の表・管理職以外は開けない） |
| OUT-04 | 設定シート＋時刻トリガー＋通し確認 | 設定シートを1枚追加し、18時トリガーを登録。営業・管理職スペースへの通知pushを含め、実配信を通しで確認 | OUT-03完了 | あり | あり（18時に実際に通知が届く・設定シートで変えられる） |

### 工程の依存

```
OUT-01（配信土台）
　↓
OUT-02（個人日報ページ）
　↓
OUT-03（日報まとめページ）
　↓
OUT-04（設定シート＋時刻トリガー＋通し確認）
　↓
日報・日報まとめ 同時リリース
```

同時リリースだが、OUT-01で土台（Webhook送信）が動くことを先に確かめる。土台が動かないまま中身を作ると、切り分けが難しくなるため。

## 8. 重要：Phase 2検索のデッドライン

**個人日報ページの初期表示を直近30日に絞るため、運用開始から31日目以降、営業は30日より古い自分の記録に画面からたどり着けなくなる。** データは deal_records に残っている（消えていない）が、見る手段が無い状態になる。

したがって、**Phase 2の検索機能（日別・店舗別・内容別）は「あれば便利」ではなく、運用開始から30日で「無いと過去に到達できなくなる」機能である。**

- **デッドライン：運用開始日から30日後。** この日までにPhase 2検索をリリースする。
- 検索は、OUT-02で作る個人日報ページ（別ページ）に、検索窓・日付選択・店舗絞り込みを足す形で実装する。ページは独立しているため、器を作り直す必要はない。
- 本デッドラインを、運用開始が確定した時点でTASK_CURRENT.mdに記録し、Phase 2の最優先項目として管理する。

## 10. この方式の利点（確認のため）

- pushで気づける（けんたろうが重視する点）。
- 日報の中身は本人だけ、まとめは管理職だけが見る（気が引ける問題が起きない）。
- Chatに商談内容が残らない（中身はpushに載せない）。入力側の思想と一貫。
- URLは全員共通でよく、管理が楽（誰が見るかはログインで決まる）。
- 個人日報を別ページにしたことで、Phase 2の検索を同じページに足すだけで拡張できる（器の作り直しなし）。
- 既存のTOP画面・ログイン・本人特定・deal_recordsをそのまま流用。抽出ロジックは日報とまとめで共通。
- DB変更は設定シート1枚のみ。新しい認証は作らない。
