# 開発工程表（営業AI音声メモ・Phase 1）

商談記録を「貯める」入力機能（Phase 1 MVP）を、どの順番で・何を作るかをまとめた作業計画。
CodexとChatGPTがこの順序どおりに実装を進めるための土台であり、けんたろうが進捗と範囲を把握するための表でもある。

- 対象：Phase 1（貯める）＝営業担当がスマホから商談を入力し、AIが仕分けしてスプレッドシートに記録するまで
- 正本との関係：この工程表は正本（要件定義書・DB設計書・API一覧・画面遷移図・技術選定理由書）から導いた作業計画である。仕様の正はあくまで正本（Word）であり、本書は正本を書き換えない
- 版：第1版（2026-07-08作成）

---

## 1. この工程表の使い方（開発の順序）

開発は「正本 → 開発工程表 → 実装指示書 → Codexが実装」の順に進む。

1. この工程表で、工程の区切り・順番・各工程の完了条件を確定する
2. 確定後、工程ごとに実装指示書を1本ずつ作る（複数工程を1本にまとめない）
3. 実装指示書ができて初めて、ChatGPTがプロンプト化しCodexが実装する
4. 各工程は、実装 → 静的チェック → 条件付き `clasp push --force` → 動作確認 → 次工程、の流れで進む。次工程へ自動で進まない

---

## 2. 全体像（9工程と依存の流れ）

前の工程が終わっていないと次が動かない、という順で並べている。

```
00 準備（設定ファイル・フォルダ・既存mdの配置）
        │
01 データの器（5つのシートを用意）
        │
        ├─ 02 入り口と本人確認（画面が出て本人を特定）
        │
        ├─ 03 AIで文章を仕分けする ─┐
        │                          ├─ 04 商談記録へ書き込む
        │                          │
        └────────────┬─────────────┘
                     │
        05 入力画面と送信（02〜04を1つの流れに束ねる。初のテスト公開・実機確認）
                     │
        ├─ 06 位置情報と店舗選び（送信の流れに店舗選びを足す）
        │
        └─ 07 完了画面（送信後の結果表示）
                     │
        08 仕上げ（公開して実機で通し確認）
```

- 03（AIで仕分け）と04（書き込み）は、05の前に単体で動くようにしておき、05で入力画面から呼び出して結合する
- 06（店舗選び）と07（完了画面）は、05の入力・送信が通ったあとに足す

## 3. 対象範囲（含むもの・含まないもの）

**このPhase 1で作るもの（含む）**

- 5つのシート：商談記録・担当者名簿・営業所名簿・店舗名簿・エラー記録
- 入力画面（S-02）と保存完了画面（S-03）
- Googleログインによる本人確認、AIによる文章の仕分け、商談記録への書き込み、位置情報からの店舗選び

**このPhase 1では作らないもの（含まない・混入させない）**

- 活用フェーズ（Phase 2）の機能：見積依頼シート・配信ログシート、商談記録のM〜R列（温度感・案件ステータス等）、API-20以降（ブリーフィング・チャット配信・横断分析等）
- 初回設定画面（S-01）：Googleログイン方式への変更により廃止済み

**正本上「MVPに含む」とされる配信3案（#13日報・#14マネージャー集約・#15朝TODO）について**

これらは蓄積したデータをGoogle Chatへ配信する機能で、入力データが貯まってから動く。したがって入力の工程00〜08を完了させたあとに、別途工程を定義して着手する（本工程表第1版の確定対象は入力の工程00〜08とする）。詳細は末尾「6. 今後の追加予定」に記載。

## 4. 各工程の詳細

各工程に、目的／作る・変えるもの／対応する正本／完了条件／確認観点／依存／人間（けんたろう）の関与、を記載する。関数名・シート名・画面名・列名は正本の正式名称を用いる。GASの物理ファイル分割（どの関数をどのファイルに置くか）は各工程の実装指示書で確定する。

### 工程00　準備

- 目的：CodexとChatGPTが作業を始めるための前提を整える。両者が読む設定・指示ファイルと、作業用の指定フォルダを用意する
- 作る・変えるもの：
  - Codex／ChatGPT用の設定・指示ファイル（AGENTS.md 他）
  - 既存の設計・運用md（DB_SCHEMA / EXPORT_SPEC / FEEDBACK_WORKFLOW_RULE / MVP_HANDOVER / PROJECT_OVERVIEW / SPEC_WEBAPP）の配置
  - 指定フォルダ（リポジトリの構成）の作成
- 対応する正本：ルールブック（運用規約）、TASK_CURRENT（作業台帳）。※開発基盤の準備であり仕様の正本には対応しない
- 完了条件：CodexとChatGPTが、決められたフォルダと設定ファイルを読んで作業を始められる状態になる
- 確認観点（この工程で見る）：設定ファイルが所定の場所にあり、フォルダ構成が想定どおりか
- 依存：なし（すべての前提）
- 人間の関与：フォルダをどこに作るか（リポジトリ構成）の確定、6つのmdの内容確認

※工程00の実装指示書を作る際に、6つのmdの中身とフォルダ構成（どこに何を置くか）の照合が必要。

---

### 工程01　データの器

- 目的：記録をためる表をすべて用意する。以降のすべての処理が読み書きする土台
- 作る・変えるもの：スプレッドシートに5つのシートを作る
  - 商談記録（deal_records）：A〜L列＋位置情報のS〜U列（緯度・経度・店舗名）。※M〜R列は活用フェーズのため作らない
  - 担当者名簿（user_master）
  - 営業所名簿（branch_master）
  - 店舗名簿（store_master）
  - エラー記録（error_log）
- 対応する正本：DB設計書 2-1〜2-7（うちPhase 1分の5シート）、R-01〜R-08のうち該当分
- 完了条件：5シートが正本どおりの列構成で存在する
- 確認観点（この工程で見る）：各シートの列名（物理名）と並び順が正本と一致するか
- 依存：工程00
- 人間の関与：スプレッドシートの目視確認。店舗名簿・担当者名簿の初期データ整備（後述）

---

### 工程02　入り口と本人確認

- 目的：URLを開くと画面が出て、ログイン中のGoogleアカウントで本人と営業所を特定する
- 作る・変えるもの：doGet（画面を返す入り口）／validateUser（ログイン者のメールで名簿照合）／lookupBranch（営業所名の引き当て）
- 対応する正本：API-01・API-02・API-10、要件定義書 FR-03、NFR-03（Googleログイン）
- 完了条件：名簿にいる担当者は画面が出て、名簿にいない・無効の担当者はエラーになる
- 確認観点（この工程で見る）：有効な担当者でログイン→本人名・営業所が特定される／未登録・無効→エラー表示
- 依存：工程01（担当者名簿を読む）
- 人間の関与：関数の実行結果確認

---

### 工程03　AIで文章を仕分けする

- 目的：送られた文章をAIに渡し、案件ごとに整理した形（構造化データ）に変換する
- 作る・変えるもの：callAiApi（AIへ送る）／buildAiRequestPayload（送信内容を組み立てる）／getSystemPrompt（AIへの指示文を返す）／getAiApiKey（APIキーを取得する）／parseAiResponse（AIの返答を解析してレコード配列にする）
- 対応する正本：API-04・API-05・API-06・API-11・API-12、要件定義書 FR-04・FR-05、AIモデルはOpenAI API
- 完了条件：サンプルの商談文が、案件（話題）ごとに正しく分かれた形に変換される
- 確認観点（この工程で見る）：複数案件が混ざった文が、案件単位で分割され、顧客名・話題・内容・宿題の項目に整理されるか
- 依存：工程01（書き込み先の項目定義に合わせる）。※画面はまだ不要。関数単体で確認
- 人間の関与：関数の実行結果確認。APIキーの設定・課金上限の再設定（後述）

---

### 工程04　商談記録へ書き込む

- 目的：整理済みの内容を商談記録シートに書く。抜けた欄は赤くし、失敗はエラー記録に残す
- 作る・変えるもの：writeRecordsToSheet（商談記録へ書く）／generateRecordId（記録IDを採番する）／markExtractionFailure（抜けを赤くし「要確認」にする）／logError（失敗をエラー記録に残す）
- 対応する正本：API-07・API-08・API-09・API-13、要件定義書 FR-06・FR-08、DB設計書（record_id 3桁ゼロ埋め・赤背景#FF0000・E/G/H列）
- 完了条件：分割された各行が正しい列に入り、必須項目の抜けは赤くなり、失敗はエラー記録に残る
- 確認観点（この工程で見る）：正しい行・列に入るか／抜けたセルが赤くなるか／記録IDが3桁連番になるか
- 依存：工程01（書き込み先）、工程03（書き込む中身）
- 人間の関与：スプレッドシートの目視確認

---

### 工程05　入力画面と送信

- 目的：営業マンが触る入力画面を作り、文字入力・音声ボタン・送信までを1つの流れに束ねる。書きかけの文章が消えない仕組みも付ける
- 作る・変えるもの：submitMemo（送信を受けて02〜04を順に呼ぶ中核処理）／S-02（商談入力画面：文字入力欄・音声ボタン・送信ボタン・未送信テキストの保全）
- 対応する正本：API-03、S-02、要件定義書 FR-01（入力・未送信テキスト保全）・FR-02（音声）。submitMemoは担当者情報をクライアントから受け取らず、サーバー側でログイン者を特定する
- 完了条件：スマホで入力 → 送信 → 商談記録に入る、までが通る（**初めてテスト公開し、実機で確認する工程**）
- 確認観点（この工程で見る）：入力→送信→記録まで通るか／音声で入力欄に文字が入るか／送信途中でブラウザを閉じても次回に文章が戻るか
- 確認観点（前工程で確認済み・再確認しない）：本人確認そのもの（工程02）、AIの仕分けそのもの（工程03）、書き込みそのもの（工程04）。※05では「束ねて通ること」を見る
- 依存：工程02・03・04
- 人間の関与：**初回のデプロイ（テスト公開）判断と、iPhone実機での動作確認**

---

### 工程06　位置情報と店舗選び

- 目的：送信時に位置を取り、近い店を4つ出して選ばせる。なければ店名を手入力し、名簿から近い名前を探す
- 作る・変えるもの：getStoreCandidates（近い順4店舗を返す）／searchStoresByName（店名の部分一致で候補を返す）／S-02の店舗選びUI（候補リスト・手入力欄）／商談記録の位置3列（緯度・経度・店舗名）への書き込み。submitMemoに店舗名・緯度・経度を受け取る引数を足す
- 対応する正本：API-14・API-15、S-02の店舗選び要素、要件定義書 FR-09、DB設計書 store_master・R-08・deal_records S〜U列。※外部の逆ジオコーディングは使わず、店舗名簿との距離照合で確定する
- 完了条件：店を選ぶと、商談記録に店名・緯度・経度が入る
- 確認観点（この工程で見る）：送信時に近い順4店舗が出るか／該当なしで手入力→名寄せ候補が出るか／選んだ店名と位置が記録に入るか
- 依存：工程01（店舗名簿）、工程05（送信の流れに足す）
- 人間の関与：iPhone実機での位置取得・店舗選びの確認。店舗名簿の初期データ整備（後述）

---

### 工程07　完了画面

- 目的：送信後に、何件に分かれたか、顧客名・話題・宿題の有無を表示し、続けて入力できるようにする
- 作る・変えるもの：S-03（保存完了画面）
- 対応する正本：S-03、要件定義書 FR-07・8-5
- 完了条件：送信後に件数と各件の概要（顧客名・話題・宿題の有無）が出て、「続けて入力する」でS-02に戻れる
- 確認観点（この工程で見る）：分割件数と概要が正しく出るか／続けて入力に戻ると入力欄が空になっているか
- 依存：工程05（送信の完了後に表示）
- 人間の関与：実機での表示確認

---

### 工程08　仕上げ

- 目的：全体を公開し、iPhoneで一通り動かして確認する
- 作る・変えるもの：デプロイ（本公開）、実機での通し確認。不具合があれば修正工程を定義
- 対応する正本：要件定義書 9-3（パイロット運用計画）、NFR-01（15秒以内）等
- 完了条件：入力 → 音声・文字 → 位置・店舗選び → 送信 → 記録 → 完了画面、の一連が実機で問題なく動く
- 確認観点（この工程で見る・回帰確認）：一連の通し動作、応答時間、複数人の同時利用
- 依存：工程05・06・07
- 人間の関与：**本公開のデプロイ判断と、パイロットメンバーへの周知、実機での通し確認**
